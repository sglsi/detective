## A-3 核心玩法总览（仅框架）

### A-3.1 核心循环（双层结构）

**v2.7 更新**：与《02血字的研究场景设计与流程》v4.0 核心循环定义对齐，从单一线性循环改为双层结构。

#### 宏观循环（案件级）

对齐《02_血字的研究》§1.3.1 宏观案件推进图：
**9 步细化展开**（项目级完整流程）：
新案件 → 承接案件 → 现场勘查 → 询问证人 → 推理验证 → 嫌疑人调查 → 抓捕凶手 → 真相揭示 → 结局

**6 阶段宏观视图**（案件级呈现，与 02 文档对齐）：
案件承接 → 现场勘查 → 证人询问 → 嫌疑人调查 → 真相揭示 → 结局评定

**说明**：9 步是完整流程的细化展开，6 阶段是案件级设计的宏观归并。两者一一对应：新案件/承接案件→案件承接；现场勘查→现场勘查；询问证人/推理验证→证人询问；嫌疑人调查/抓捕凶手→嫌疑人调查；真相揭示→真相揭示；结局→结局评定。案件级设计文档统一使用 6 阶段表述。

#### 微观循环（证据级）——「主动发现」六步探索闭环

对齐《02_血字的研究》§4.3 六步探索闭环：
观察发现 → 工具操作 → 数据记录 → 知识检索 → 假设形成 → 验证修正
    ↑                                                        │
    └──────────────── 错误时返回重新探索 ─────────────────────┘

**关键设计原则**：
• **双层嵌套**——宏观循环是案件骨架，微观循环是每一次证据发现的肌肉；玩家在宏观的每个阶段中反复执行微观六步闭环
• **主动发现**——玩家是探索的驱动者，Step 1-3（观察发现、工具操作、数据记录）由玩家主动完成，非系统自动触发
• **玩家选择权**——Step 4-5（知识检索、假设形成）是系统提供的辅助功能，玩家可自主选择是否使用，不强制
• **不惩罚原则**——错误推理可推翻重做不扣分，过度探索不扣分；微观循环中错误不是终点，而是探索的一部分
• **循环奖励**——每次微观循环都会积累经验值，解锁新的观察点和工具，发现新的知识关联

### A-3.2 玩法定位

• **不是小说复述**——玩家不能通过读对话看剧情就过关
• **不是问答题**——线索不会以明显形式"提示"玩家去点
• **是侦探模拟**——玩家通过主动观察环境、操作工具、记录数据、检索知识、形成假设、验证假设来推进；系统不代劳，玩家是探索的驱动者

### A-3.3 三大沉浸支柱

1. **历史真实性**——1864 年斯坦福图书馆版伦敦真实地图 + 时代细节
2. **推理可玩性**——证据可发现可关联可修正
3. **判断有压力**——时间推进、误导存在、评分激励

**本节为 v2.3a 新增，v2.7 更新**——基于《02血字的研究场景设计与流程》v4.0 核心循环定义，5 项能力名称与 §1.4 对齐。

**核心原则**：**软锁而非硬锁**——时间/时效锁是设计意图，不是惩罚机制。**任何进度锁都至少有 1 种"绕过去"和 1 种"补救"方法**——保证玩家永远不会因进度锁陷入"无路可走"的死局。

#### B-5.5.1 时间锁的设计定位

• **时间锁是什么**：剧情/事件上的"窗口"——某些事件只在某段进度内可触发

• **核心目的**：让玩家感受到"调查是有节奏的"，而不是"卡住后无法继续"

#### B-5.5.5 死局检测（**反例**）

以下情况属于"设计死局"，**必须**在工单审阅时发现并修复：
1. ❌ 进度锁事件**既无绕过去方法也无补救方法**——玩家卡死后无任何路径
2. ❌ 绕过去方法本身也被进度锁——"绕过去的绕过去也被锁"
3. ❌ 补救方法只在"玩家已知信息"前提下可触发——但玩家根本不知道这条线索
4. ❌ 进度锁事件完成后，**无任何下游提示**告诉玩家"还有补救入口"
5. ❌ 案件多个进度锁事件连环锁——任一卡死即全盘卡死
**检测方法**：
• 案件工单审阅时**强制检查**所有定义了 lock_type 事件的 bypass_methods + remedy_methods 字段
• 自动化校验：见 D-24.5 双向锁机制
**→ 详细规则**：参见 02_场景设计文档 §3 调查进度系统

### B-10.6 推理错误记录机制（v4.0 三星评价适配）

**本节为 v4.0 更新**——基于《02血字的研究场景设计与流程》v4.0 三星评价体系重写。
**设计原则**：错误是推理的自然过程，不应惩罚。**推理错误不扣分、不衰减、不关闭推理链**。
**核心机制**：
• **推理结论只作为玩家推理过程的记录**：玩家提交推论后，无论正确还是错误，都会被记录在推理墙的"推断"页签
• **错误推论允许在后续继续修改**：玩家可在任何阶段返回"推断"页签修改、删除或重新提交推论
• **错误不导致推理链失效**：错误推论不会让对应的推理链判定为"失败"，推理链依然保持开放、可继续推进
• **错误不扣分、不衰减**：错误推论不扣减星级，不出现在三星评价公式中
**错误次数记录（保留展示用途）**：
• **记录方式**：系统为每条推理链独立记录错误次数（reasoning_chain_error_count: Dict<chain_id, int>）
• **用途**：仅用于推理墙"推断"页签展示 + 能力徽章"锲而不舍"判定（同一推理链尝试 3 次后成功）
• **不影响星级**：推理之星仅取决于**最终提交时的正确判断比例**，历史错误次数不参与星级计算
→ 详细规则：参见《02血字的研究场景设计与流程》§4.5《推理提交与星级评价（v4.0 三星评价版）》+ §7《评价与徽章系统（v4.0 三星+徽章版）》

### B-11.5 三星评价公式（v4.0 三星评价适配）

**本节为 v4.0 更新**——基于《02血字的研究场景设计与流程》v4.0 三星评价体系重写，替代原 v2.5"推理链计分公式 ADR"。
**v4.0 重大变更**：从"五系数连乘数值公式"调整为"三维独立离散星级"。
**设计原则**：
• 评价维度从"单一数值"变为"三维独立"——让玩家清楚知道自己在观察、推理、洞察三个方向上的表现
• 错误不扣分、不衰减——鼓励试错，与福尔摩斯式探案"试错中逼近真相"的本质一致
• 即时反馈——每次提交推理后立即显示星级，玩家可立即知道哪些维度可提升
**三星评价公式**：
**单条推理链总评 = 观察之星 + 推理之星 + 洞察之星**（三维独立，各 1-3⭐）

**单条推理链最高星级**：3 维 × 3⭐ = **9⭐**
**案件总满星**：14 条推理链 × 9⭐ = **126⭐**
**满星条件（同时满足）**：
1. 该推理链所有观察事实+中间判断全部收集（缺失 0 条 → 观察星 = 3⭐）
2. 所有中间判断均正确（推理星 = 3⭐）
3. 按关键→重要→一般顺序调查（洞察星 = 3⭐）
**徽章等级（与难度无关，由星级评定决定）**：
徽章等级仅由单条推理链的总星级决定，三种难度模式均可获得铜/银/金徽章。详细对应关系见 §B-11.2。

• **存档字段**：每条推理链需记录 observed_facts_collected（观察事实收集列表）、intermediate_deductions_collected（中间判断收集列表）、intermediate_deductions_correct（正确判断列表）、reasoning_chain_direction_log（方向选择历史）、hint_usage（提示使用记录）
• **错误次数记录**：reasoning_chain_error_count 仅用于推理墙展示和专项徽章判定，不参与星级计算
• **数据结构**：Dict<chain_id, ChainProgress>，每条推理链独立维护进度
→ 详细规则：参见《02血字的研究场景设计与流程》§4.5《推理提交与星级评价（v4.0 三星评价版）》+ §7《评价与徽章系统（v4.0 三星+徽章版）》

## D-23 版本管理

**作用**：定义版本号规则、固化边界、迭代节奏、废弃流程（**未固化**——待实践调整）。

### D-23.1 双版本号体系（**v2.3 新增**）

**本节为 v2.3 重大重写**——原 v2.0~v2.2 仅有"游戏客户端版本号"三段式。v2.3 引入**双版本号**：游戏客户端版本号 + 内容数据版本号。
**为何需要双版本号**：
1. **内容更新和 Bug 修复的发布周期不同**——内容数据（新增案件、新增 NPC 立绘）应能独立于客户端升级
2. **避免一次小修小补要玩家重新下载整个客户端**——内容数据可走 CDN 增量更新
3. **M3+ 跨案件时，案件内容可作为"内容数据"独立分发**——支持 A/B 测试、新案件灰度发布

#### D-23.1.1 游戏客户端版本号（**引擎/功能版本**）

采用三段式：主版本.次版本.修订号

**特殊版本号**：
• 1.0.0-alpha：内部测试版
• 1.0.0-beta：公开测试版
• 1.0.0-rc：发布候选
• 1.0.0：正式发布
**MVP 节奏**：客户端 v1.0.0（MVP 上线）→ v1.0.X（修订号）→ v1.1.0（M3 完整版）→ v2.0.0（M4 扩展/移动端）

#### D-23.1.2 内容数据版本号（**案件/资源版本**）

采用独立三段式：主版本.次版本.修订号，但**与客户端版本号解耦**

**特殊版本号**：
• case_data: 1.0.0-sotcb-rc1：内容数据 + 案件代号 + 候选
• asset_bundle: 1.0.0-baker-street-rc1：资源包 + 场景代号 + 候选
**MVP 节奏**：
• case_data: 1.0.0（《血字的研究》上线）→ 1.0.X（修订号）→ 1.1.0（第二个案件《四签名》）

#### D-23.1.3 双版本号关系与发布策略

**核心原则**：**双版本号可独立升级，但必须保证兼容性**

**兼容性规则**：
1. **客户端向下兼容内容数据**——v1.0.0 客户端必须能加载 case_data 1.0.X（修订号兼容）
2. **客户端主版本可拒绝旧内容数据**——v2.0.0 客户端可拒绝 case_data 1.0.0（提示玩家更新内容数据）
3. **内容数据主版本升级需要客户端支持**——case_data 2.0.0 仅在客户端 v1.1.0+ 上加载

| 核心循环双层结构 | 游戏的核心玩法循环分为两层：宏观循环（案件级）——承接案件→现场勘查→询问证人→推理验证→破案结案，贯穿整个案件生命周期；微观循环（证据级）——主动发现六步闭环（观察环境→发现异常→放大检查→收集碎片→组合线索→形成判断），在单个证据层面反复执行。两层循环嵌套运行，微观循环为宏观循环提供推理素材，宏观循环为微观循环提供叙事框架。 | A-3.1 |
| 双向锁机制 | 确保 L1/L2/L3 三层文档一致性的校验机制，包含两个方向：正向锁——L1 规范字段变更后，自动检查所有引用该字段的 L2/L3 文档是否同步更新；反向锁——L3 工单中使用的字段/术语必须在 L1 中有定义，若出现未定义字段则触发告警。通过 Git Hook / CI 流程在合并时自动校验引用有效性，防止文档间出现不一致。 | D-24.5, 0.4 R-5 |
| 福尔摩斯级评价 | 游戏中最高级别的评价等级，代表玩家以接近福尔摩斯本人的推理水平完成案件。获得条件：在困难模式下，所有推理链均达到 3 星满星（观察/推理/洞察各 3⭐），且发现全部隐藏细节。该评价不仅是数值达标，更代表玩家展现了完整的演绎推理能力——从观察细微线索、构建假设、验证矛盾到最终破案的全流程无瑕疵表现。 | B-11.2, §7.2 |
| 三层文档体系 | 项目文档的分层架构：L1 规范层（本目录）定义"写什么、用什么字段、到哪查"；L2 设计层（如 02血字的研究场景设计、08系统框架设计）定义系统架构和交互逻辑；L3 工单层（如 03_关卡设计稿）定义具体执行细节。上层约束下层，下层反馈上层，修改需按层级传导。 | 0.1 |
| 主动发现六步闭环 | 微观循环的核心执行流程：①观察环境→②发现异常→③放大检查→④收集碎片→⑤组合线索→⑥形成判断。其中 Step 1-3 为必做步骤（系统保证），Step 4-5 为可选步骤（玩家主动探索）。该闭环在每个可交互对象上独立执行，确保玩家始终处于"发现-验证"的正反馈循环中。 | A-3.1, §B-9 |
| 软锁 vs 硬锁 | 时间锁的两种模式：软锁（soft）——时间窗口错过后有替代路径或补救措施，玩家不会卡死；硬锁（hard）——严格的时间窗口，错过则永久失去该内容。本项目强制要求所有时间锁必须为软锁，且至少提供 1 种绕过去方法和 1 种补救方法。 | B-5.5 |
| 推理链三段式结构 | 每条推理链的标准结构：观察事实（玩家在场景中收集到的客观数据）→中间判断（基于观察事实的逻辑推导）→最终结论（指向案件真相的关键判定）。三段式结构确保推理过程可追溯、可评分，是推理墙系统和星级评价的基础。 | B-10.5 |
| 案件工单 | 单个案件的 L3 级完整执行文档，包含 15 个标准模块（案件概述、主线阶段、场景工单、线索表、推理节点表、评分规则等），是程序/美术/UI/音频/测试的协作依据。每份案件工单必须严格遵循 L1 规范，并通过一致性校验。 | C-13 |

# 2. E-25.1 引擎与框架

本章为  的展开版本。技术选型详细对比见 ；分层架构与 Autoload 表见 。

## 2.1 引擎版本锁定

**版本锁定说明**：已确认使用 Godot 4.7 作为锁定版本。详见 ADR-#1。

## 2.2 脚本语言分工

**GDScript 优势**（按 TOOLS.md 实战经验排序）：① 类 Python 语法，与 Godot 深度集成无需编译；② 迭代极快，改代码 F5 立刻生效；③ 与 Resource/Node/Signal 粘合度最高；④ 移动端导出无需 .NET 运行时，包体更小。
**C# 引入边界**：① 仅在 GDScript 性能不达标（>16ms/帧 的热路径）时引入；② C# 模块必须通过 Signal/事件总线与 GDScript 通信；③ C# 编译产物随版本提交（或用 CI 缓存），避免"换机器跑不起来"。

## 2.3 渲染管线选型

**Compatibility (GLES3)** 作为唯一渲染管线，理由：
• Web（HTML5）导出必须用 GLES3（WebGL2）
• 覆盖 Android 5.0+ / iOS 11+ 全部目标设备
• 2D 性能与 Forward+ 在本项目分辨率（1920×1080）下无明显差异
• 避免"两套着色器代码"
**不引入 Forward+ / Mobile** 的理由：2D 推理探案游戏无 PBR / 阴影 / 后处理需求，Forward+ 的优势用不上；Mobile 渲染器在 Web 端兼容性差。

## 2.4 数据格式分层

**为什么内容用 Resource 不用 JSON**：Resource 继承 RefCounted 可挂自定义类型、IDE 补全友好；自带 @export 编辑器面板，关卡设计师在 Godot 编辑器直接调字段；序列化带类型信息；支持引用嵌套（如 ClueState 引用 CaseState），JSON 需手写 ID 引用。
**Resource 的局限**（ADR-#4）：二进制 Resource 不可手工 diff。**解决方案**：游戏内容全部提交 .tres（文本格式），仅玩家存档用二进制。

### 5.2.1 游戏全局状态机（GameStateMachine）

[BOOT] ─→ [MENU] ─新游戏─→ [LOADING_CASE] ─→ [IN_CASE]
  ↑                                       ↓ ↑
  └──── [CREDITS] ←── [CASE_COMPLETE]   [PAUSE] (Esc)
           ↑                                  ↓

### 5.2.2 对话状态机（DialogueStateMachine）

[IDLE] → [TYPING] → [WAIT_INPUT] → [NEXT_LINE] → [WAIT_INPUT]
                          ↓
                   [CHOICE_PRESENT] → [BRANCH] → [TYPING]
                          ↓
                   [SHOWING_TOOLTIP] → [WAIT_INPUT]
                          ↓  对话结束
                         [END] → [IDLE]

**关键状态**：TYPING（打字机效果可"快进"）、WAITINPUT（等待点击）、CHOICEPRESENT（2~4 个分支选项）、BRANCH（按选择跳转节点）、SHOWING_TOOLTIP（线索/物品提示框）。

### 5.2.3 调查/推理状态机（InvestigationStateMachine）

[EXPLORING] ─点击热点─→ [OBSERVING] ─发现线索─→ [CLUE_FOUND]
       ↑                      │
       │                  [无新线索]
       └────────────── BACK_TO_EXPLORING ─┘

[EXPLORING] ─ 推理墙触发 ─→ [PUZZLE_MODE] ─→ [WAITING_INPUT]
                                              ↓ 玩家提交
                                          [JUDGING]
                                              ↓
                          [CORRECT]/[PARTIAL]/[WRONG] → BACK_TO_PUZZLE_MODE
                          [HINT] → WAITING_INPUT

**关键状态**：EXPLORING（自由探索，点击热点）、OBSERVING（播放观察动画）、CLUE_FOUND（线索入袋，触发 clue_discovered 事件）、PUZZLE_MODE（推理墙，整理线索）、JUDGING（系统判定对错，CORRECT/PARTIAL/WRONG 三档反馈，详见下方 §5.2.4 难度模式）。

### 5.2.4 难度模式

游戏提供三档难度，影响推理墙的交互方式和提示机制：

**实现要点**：
• 难度选择入口：开始案件前 / 设置中随时切换
• 难度配置存储于 config.json 的 difficulty 字段，运行时由 DifficultyManager 管理
• 提示概率函数（普通模式）：hint_probability = min(0.8, elapsed_stuck_time / 300)，每 5 分钟递增 10% 概率触发提示

# 6. 网络与后端架构

本章详细展开 ADR-#5、ADR-#9、ADR-#11、ADR-#13。Supabase 选型详细对比见 。

## 6.1 Supabase 选型理由

**选择 Supabase 的核心理由**：① **关系型数据**——案件/线索/对话/用户进度强关系，PostgreSQL 比 NoSQL 更适合；② **免费额度够 MVP**——500MB 数据库足够 10 万用户；③ **开放格式**——SQL 导出便利，避免厂商锁定；④ **Realtime**——可实现跨设备实时同步（MVP 阶段可选）。
**自建后端备选**（M3+ 评估）：若 Supabase 免费额度不够或需要特殊合规（如国内 ICP 备案），可迁移到自建 Node.js + PostgreSQL。数据模型层预留抽象，业务层调用 APIManager 不直接耦合 Supabase。

## 6.2 数据模型（PostgreSQL Schema）

MVP 阶段 3 张核心表（user_profiles + player_saves + analytics_events）；成就/徽章表 v2.0 启用。
**// sql
**-- 用户档案（与 auth.users 一对一）
CREATE TABLE public.user_profiles (
    id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,

-- 玩家存档（每个用户每个案件一行）

# 8. 安全与合规

## 8.1 存档加密

### ADR-#13: CDN 与静态资源分发

**决策内容**：**方案② 腾讯云 EdgeOne 免费（国内主分发）+ Cloudflare 免费（海外备份）**双轨架构。

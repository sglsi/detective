 # 技术工具指南（TOOLS.md）

 ## 一、核心技术选型

 ### 1. 游戏引擎

 **Godot 4.7**（锁定版本号 4.7，避免多开发者引擎版本不一致导致缓存文件冲突）
 - 渲染器：Compatibility / GLES3（Web 导出兼容性最佳，GLES3 在桌面端有更好性能）
 - Web 导出：第一阶段网页游戏发布
 - 原生导出：第二阶段手机应用（Android / iOS）
 - 脚本语言：**GDScript**（类 Python，与引擎深度集成）
 - 地图系统：**TileMapLayer**（Godot 4.3+ 原生瓦片地图）

 > 建议在项目根目录放置 `.godot-version` 文件并写入 `4.7`，标注锁定版本，避免多开发者引擎版本不一致导致的 `uid_cache.bin` / `global_script_class_cache.cfg` 冲突。

 ### 2. 前后端通信

 | 组件 | 用途 | 引入阶段 |
 |------|------|----------|
 | `HTTPRequest`（Godot 内置节点） | 所有业务 CRUD API 调用，异步信号驱动 | M1（MVP）|
 | WebSocket（`WebSocketPeer`） | 线索实时推送、跨设备同步通知 | M2+ |

 **阶段说明：**
 - **M1（MVP）**：仅使用 `HTTPRequest`，不做 WebSocket。MVP 核心是验证"推理体验是否有效有趣"，通信层越简单越好。
 - **M2+**：引入 WebSocket 用于「线索实时推送」（如同步另一终端的新发现），以及可选的在线状态通知。
 - **M3+**：若涉及多人协作推理，WebSocket 升级为基础发送通道。

 **注意：** `HTTPClient`（底层同步 API）不在本项目中选用。它需要手动管理请求队列、连接池、重试、Token 刷新——相当于手写一个 HTTP 框架，性价比低。业务层直接使用 `HTTPRequest` 的 `request_completed` 信号即可满足需求。

 ### 3. 后端统一 API

 Web 端和移动端共用同一后端接口，保证数据一致性。API 风格推荐 RESTful，请求/响应体使用 JSON。

 ### 4. 存储方案

 - **游客模式**：本地临时缓存（`Storage` / `FileAccess`），进度不持久化，退出即丢失。
 - **注册用户**：本地存档 + 后端服务器同步，支持跨设备、跨平台进度同步。

 ## 二、关键系统实现指南

 ### 1. 注册与游客系统

 - **游客模式触发**：玩家打开游戏时默认进入游客模式，进度仅保存在浏览器本地缓存。
 - **注册触发时机**：当游客尝试保存进度时，自动弹出注册窗口。
 - **注册验证机制**：
   - 用户名注册：仅需验证密码强度
   - 手机号注册：发送短信验证码验证，确保真实性
   - 邮箱注册：发送邮箱验证码验证，确保可找回
 - **跨平台同步**：同一账号在 Web 端和移动端登录时，自动同步所有进度数据。

 ### 2. 存档同步冲突策略

 当同一账号在多设备上产生分歧存档时，同步策略如下：

 | 阶段 | 策略 | 说明 |
 |------|------|------|
 | M1（MVP） | **服务端覆盖** | 以服务端存档为权威（最新时间戳为准），本地存档直接替换。简单粗暴，避免冲突合并的复杂度。 |
 | M2+ | **案件级合并** | 不同案件取并集；同一案件的线索按时间戳逐条合并。 |
 | M3+ | **双向合并** | 服务端 + 本地取并集，冲突线索提供手动选择界面。 |

 **原则：** M1 阶段不做冲突合并，MVP 的核心是验证推理体验，而非同步一致性。

 ### 3. 案件驱动式瓦片加载

 每个案件定义关联瓦片的最小包围矩形（`Rect2`），仅加载必要区域。新案件区域与已有探索区域叠加扩展，已探索区域永久保留。迷雾系统仅覆盖未到达的区域。

 **实现细节：**
 - **分层策略**：使用「单个 `TileMapLayer` + 区域坐标管理」，而非按案件分 Layer。案件切换时，通过 `set_cell()` 动态增删瓦片。
 - **过渡策略**：案件切换时使用「淡入」过渡（`modulate.a` 从 0 → 1，持续 0.3s），视觉上掩盖瓦片加载延迟。使用 `AnimationPlayer` 或 `Tween` 实现。
 - **迷雾系统**：使用 `CanvasGroup` + 子 TileMapLayer 覆盖迷雾区域，配合鼠标悬停/走过驱散效果。

 ### 4. 多难度模式实现

 - **简单模式**：自动填写线索墙，动态小说式推进，提供明确指引
 - **普通模式**：平衡推理与提示，根据玩家进度调整提示概率，防流失
 - **困难模式**：硬核推理，线索墙需手动填写，仅提供底线指引

 ## 三、技术原则遵循

 ### 1. 稳定性原则

 - 选用稳定、成熟的技术架构，避免使用实验性功能
 - 移动端适配优先考虑兼容性，覆盖大部分 Android 设备
 - 资源预加载和缓存机制，减少卡顿和崩溃

 ### 2. 可扩展性原则

 - 模块化设计，功能模块解耦，便于后续扩展
 - 案件库架构支持动态添加新案件，无需底层重构
 - 数据驱动设计，配置文件与代码分离，便于调整参数

 ## 四、项目目录结构约定

 以下为 `res://` 下的推荐目录骨架，所有模块均按职责组织：

 ```
 res://
 ├── scenes/          # 场景文件（.tscn）
 │   ├── systems/     # 系统级场景（存档管理、设置、主菜单）
 │   ├── cases/       # 案件相关场景
 │   └── ui/          # UI 场景（HUD、线索墙、地图）
 ├── scripts/         # GDScript 脚本（.gd）
 │   ├── systems/     # 系统逻辑
 │   ├── cases/       # 案件逻辑
 │   └── ui/          # UI 逻辑
 ├── cases/           # 案件数据文件（.tres / .res）
 ├── assets/          # 资源文件（图片、音频、字体）
 │   ├── tilesets/    # 瓦片集
 │   ├── sprites/     # 精灵图
 │   └── fonts/       # 字体
 ├── data/            # JSON 配置文件（对话、线索表、参数配置）
 │   ├── dialogue/    # 对话数据
 │   └── config/      # 系统参数配置
 └── addons/          # 第三方插件
 ```

 **约定：**
 - 所有函数参数和返回值加类型注解（GDScript 静态类型），提升可维护性
 - 案件数据使用 `.tres`（Godot Resource 文件）或 JSON 格式存储——`.tres` 天然可被 `@export` 在编辑器中编辑，JSON 适合后期外部工具生成
 - 配置文件和代码分离，避免硬编码

 ## 五、标准化目录方法论

 ### 1. 三层文档体系协作

 - **L1 → L3 引用方向**：L1 规范文档定义字段名、命名规范；L3 案件工单直接用 L1 字段表
 - **L3 → L1 反向引用**：每个 L3 文档头部加「📍规范引用」块，指向 L1 的具体节号
 - **修改同步规则**：
   - A 类固化字段 → L1/L3 全项目同步
   - B 类增量 → L1/L3 同步
   - C 类案件级 → 案件内即可

 ### 2. MVP 简化原则

 - **核心交互优先**：先验证玩家核心交互是否"有效、有趣"，再考虑体验升级
 - **关联图谱放 M3+**：推理墙基础态（5 页签 + 拖拽 + 分组）MVP 阶段必须；关联图谱放 M3/M4
 - **豁免清单思维**：M1/M2 可豁免项必须有替代方案 + 明确补全时机，避免"为规范而规范"

 ### 3. 软锁而非硬锁

 - **时间锁必带绕过去 + 补救**：避免玩家陷入"无路可走"的死局
 - **死局检测**：案件工单审阅时强制检查所有 `time_lock=true` 事件的 `bypass_methods` + `remedy_methods` 字段

 ### 4. 双版本号解耦

 - **客户端版本号（vX.Y.Z）** = 引擎 / 功能变更
 - **内容数据版本号** = 案件 / 资源变更（独立编号）
 - **存档格式版本号（save_version）** = 与前两者独立，可独立升级
 - 三者独立才能真正解耦"修 Bug"和"加内容"的发布周期

 ### 5. 双向锁校验（Git Hook / CI）

 - 合并关卡稿到主分支时自动运行校验器
 - 校验内容：引用的本目录节号是否仍有效
- 目的：避免"修了字段名忘了改引用"导致的连锁错误

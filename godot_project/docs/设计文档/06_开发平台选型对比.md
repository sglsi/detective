# 开发平台选型对比分析

> 生成时间：2026-06-02
> 项目定位：维多利亚伦敦探案游戏（2D推理探案，手机端为主，前后端通信，广告变现）

## 一、候选平台

基于项目特征（2D、UI密集、手机优先、需要服务端通信），筛选出4个候选方案：

| 平台 | 类型 | 语言 | 许可证 |
|------|------|------|--------|
| **Godot 4.x** | 游戏引擎 | GDScript / C# | MIT（完全免费） |
| **Unity** | 游戏引擎 | C# | Personal免费/Pro $40/月 |
| **Web + Capacitor** | Web套壳 | TypeScript / React | 开源 |
| **Cocos Creator** | 游戏引擎 | TypeScript | MIT引擎/商业服务收费 |

## 二、逐项对比

### 2.1 开发成本与许可

| 维度 | Godot 4.x | Unity | Web + Capacitor | Cocos Creator |
|------|-----------|-------|-----------------|---------------|
| 引擎费用 | **完全免费**，无版税 | Personal版免费（年收入<10万美元）；Pro $40/月 | 免费 | 引擎免费，部分云服务收费 |
| 版税 | 无 | 无 | 无 | 无 |
| 隐性成本 | 插件少，部分SDK需自己写原生桥接 | Asset Store插件可能产生费用 | Capacitor插件生态免费为主 | 国内SDK集成方便，海外SDK需适配 |
| 适合阶段 | **一个人独立开发，零成本启动** | 有预算或团队时更合适 | 已有Web代码时低成本扩展 | 中小团队2D手游 |

**判断：** Godot成本最优，Web+Capacitor次之（复用现有Web MVP代码）。

### 2.2 2D游戏能力

| 维度 | Godot 4.x | Unity | Web + Capacitor | Cocos Creator |
|------|-----------|-------|-----------------|---------------|
| 2D渲染 | **专用2D引擎**，像素坐标，无伪3D折衷 | URP 2D可用，但本质是3D相机的正交投影 | Canvas/WebGL渲染，Phaser/原生DOM均可 | **2D原生优化**，spine动画支持好 |
| UI系统 | Control节点系统，内置丰富UI组件 |UGUI较成熟，但2D UI布局偏复杂 | **DOM+CSS最灵活**，React生态无敌 | 内置UI组件，Creator编辑器拖拽布局 |
| 动画系统 | AnimationPlayer + Tween，轻量够用 | Animator + Timeline，功能强大但偏重 | CSS动画 + JS Tween，灵活但性能受限 | DragonBones/Spine + 内置动画编辑器 |
| 地图集成 | 需自行实现地图渲染（TileMap可用） | Asset Store有地图插件 | **Leaflet.js原生支持**（已验证可行） | 内置TiledMap支持 |

**判断：**
- 纯2D能力：Cocos ≈ Godot > Unity > Web（但差距不大）
- UI开发效率：Web（DOM/CSS）> Cocos ≈ Unity > Godot
- 地图系统：Web（Leaflet.js）远超其他三个

### 2.3 移动端表现

| 维度 | Godot 4.x | Unity | Web + Capacitor | Cocos Creator |
|------|-----------|-------|-----------------|---------------|
| Android导出 | ✅ APK/AAB直接导出 | ✅ 成熟导出流程 | ✅ Capacitor打包AAB | ✅ 原生导出 |
| iOS导出 | ✅ 需Mac+Xcode | ✅ 成熟导出流程 | ✅ Capacitor打包 | ✅ 原生导出 |
| 包体大小 | **~5-15MB**（极小） | 20-50MB起步 | 10-30MB（含WebView） | 8-20MB |
| 低端机兼容 | Compatibility渲染器覆盖广 | URP可配置低端 | **WebView性能受限**，中低端机卡顿 | 引擎轻量，2D兼容性好 |
| 触控交互 | Input系统完善 | Input System完善 | 需自行处理touch事件 | 内置触控事件系统 |
| 热更新 | 需自建或第三方方案 | 需自建 | **Capgo等方案成熟** | 引擎内置热更新支持 |

**判断：**
- 包体与兼容性：Godot最优，Cocos次之
- 热更新：Web+Capacitor最成熟，Cocos有内置方案
- 低端机：Godot（Compatibility模式）≈ Cocos > Unity > Web+Capacitor

### 2.4 服务端通信

| 维度 | Godot 4.x | Unity | Web + Capacitor | Cocos Creator |
|------|-----------|-------|-----------------|---------------|
| HTTP请求 | HTTPRequest节点 | UnityWebRequest | **fetch/axios原生** | XMLHttpRequest/axios |
| WebSocket | 内置WebSocket | 内置+第三方 | **原生WebSocket API** | 内置WebSocket |
| 数据序列化 | JSON（需手动parse） | JsonUtility | **原生JSON支持** | 原生JSON支持 |
| 离线存储 | FileAccess + ConfigFile | PlayerPrefs + 文件 | **LocalStorage/IndexedDB/SQLite** | localStorage + 文件 |
| 加密/认证 | 需自实现或插件 | 需插件 | **Web Crypto API** | 需自实现 |

**判断：** Web+Capacitor在服务端通信上优势明显——所有Web API原生可用，开发效率最高。Godot和Unity需要额外的网络层封装。

### 2.5 广告变现SDK

| 维度 | Godot 4.x | Unity | Web + Capacitor | Cocos Creator |
|------|-----------|-------|-----------------|---------------|
| AdMob | 社区插件（非官方） | **官方AdMob包** | Capacitor插件 | 官方/社区插件 |
| AppLovin | 需原生桥接 | **官方支持** | Capacitor插件 | 社区插件 |
| Pangle（穿山甲） | 需自己写GDExtension | 有第三方包 | Capacitor插件 | **官方/社区支持好** |
| IAP（内购） | 需插件 | **Unity IAP成熟** | Capacitor RevenueCat | 内置IAP |

**判断：** Unity广告SDK生态最成熟；Cocos在国内SDK（尤其穿山甲）支持上更好；Godot和Web+Capacitor需要更多原生桥接工作。

### 2.6 开发效率与生态

| 维度 | Godot 4.x | Unity | Web + Capacitor | Cocos Creator |
|------|-----------|-------|-----------------|---------------|
| 学习曲线 | **GDScript类Python，上手快** | C# + 编辑器较复杂 | 前端开发者零成本 | TypeScript，前端友好 |
| 编辑器 | 轻量（~100MB），启动秒开 | 重型（数GB），启动慢 | VSCode + 浏览器调试 | 中型，有可视化编辑器 |
| 迭代速度 | **即改即跑，无编译等待** | 编译等待较长 | HMR热更新，极快 | 预览较快 |
| 社区/教程 | 快速增长中，但2D手游案例偏少 | **最丰富的教程和资源** | Web社区庞大，游戏向偏少 | 国内社区活跃，海外较少 |
| Asset Store | 刚上线（2026-05），内容少 | **最大资产商店** | npm生态无限 | 有自己的商店 |
| 调试工具 | 内置调试器+远程调试 | Profiler强大 | **Chrome DevTools** | 内置调试器 |

**判断：**
- 上手速度：Godot ≈ Web > Cocos > Unity
- 迭代效率：Web（HMR）> Godot > Cocos > Unity
- 资源丰富度：Unity >>> Cocos > Godot > Web

## 四、推荐方案

### 首选：Godot 4.x（Web导出优先，原生移动端跟进）

**理由：**

1. **零成本+零版税**，一个人做独立游戏最现实的起点
2. **专用2D引擎**，你的游戏没有3D需求，Godot的2D坐标系统比Unity的正交投影更直觉
3. **支持Web导出**，第一阶段直接导出为网页游戏，玩家打开即玩，降低进入门槛
4. **同一引擎覆盖Web+移动端**，无需维护两套代码，Web导出和原生导出共享全部GDScript代码
5. **包体极小**（5-15MB），适合休闲游戏用户快速下载
6. **低端机兼容好**，Compatibility渲染器覆盖极广的Android设备
7. **迭代极快**，改完代码按F5直接跑，没有编译等待——对独立开发者极其重要
8. **MIT许可证**，没有Unity 2023年那种授权条款翻车的风险

**分阶段发布策略：**
- **第一阶段（网页游戏）**：Godot导出HTML5/Web，玩家直接在浏览器中打开即玩，支持游客体验+注册保存
- **第二阶段（手机应用）**：网页端玩家积累到一定数量后，Godot导出Android/iOS原生应用，共用同一后端

**主要短板与应对：**

| 短板 | 应对方案 |
|------|---------|
| 广告SDK少 | 写GDExtension桥接AdMob/AppLovin原生SDK，一次封装永久复用；社区已有AdMob插件可参考 |
| 地图系统需自建 | 用TileMap + 自定义地图渲染层替代Leaflet.js；地图数据用JSON/Tiled格式导入 |
| 网络层需封装 | 封装HTTPClient + WebSocket单例，JSON序列化写工具类 |
| Web导出性能 | 2D推理游戏计算量不大，Web导出性能足够；如遇瓶颈再优化或引导玩家使用移动端 |
| 社区案例少 | 2D手游案例虽少，但Brotato、Buckshot Roulette等已验证商业可行性 |

### 备选方案：Web + Capacitor（过渡路线）

如果Godot Web导出的性能或体验达不到要求，可以：

1. **MVP阶段**用 Web（React + Leaflet.js）快速验证核心玩法
2. **手机端**用 Capacitor 打包上架，复用全部Web代码
3. **正式版**评估性能瓶颈后，再决定是否迁移到Godot

**优势：** 地图系统零迁移成本，Web开发效率最高，服务端通信最自然
**风险：** 低端机WebView性能差，广告SDK桥接工作量大，长期维护两套代码

### 不推荐Unity的理由

- 你的游戏是纯2D推理探案，没有3D渲染需求，Unity的重量级架构是过度工程
- Personal版有收入限制（<10万美元），超过后必须付费
- 包体大（20-50MB起步），对休闲游戏用户不友好
- 2023年的授权条款风波证明商业风险不可控

### 不推荐Cocos Creator的理由

- 海外SDK生态弱于Unity/Godot，你的广告方案含AdMob+AppLovin等海外平台
- 编辑器稳定性口碑一般，社区反馈常有"莫名报错"
- 如果不做国内市场（穿山甲为主），Cocos的核心优势发挥不出来
- TypeScript生态虽好，但Godot的GDScript学习成本更低且与引擎深度优化

## 七、基于《00》《02》最新设计的重新评估

> 补充时间：2026-06-29
> 背景：原文档（2026-06-02）基于通用 2D 推理游戏评估。随着《00_标准化开发目录》v2.9 和《02_血字的研究_场景设计与流程》v3.5.0 重构出的 **"主动发现六步闭环"** 和 **"推理墙系统"**，对引擎底层提出了更特定的要求。本章基于最新设计进行重新评估。

### 7.1 新设计对引擎的底层要求

Godot 对这个项目的核心价值在于：**2D 像素坐标零误差**（脚印间距计算）、**AnimationPlayer 驱动任意属性**做镜头演出、**Control 原生拖拽**做推理墙、**RichTextLabel+BBCode** 做知识库高亮。以下逐项展开：

| 最新设计核心点（来自 L1/L3） | 对引擎的底层要求 | Godot 4.x 的表现 | 其他引擎的致命伤 |
|:---|:---|:---|:---|
| **1. 推理墙 UI**（5页签拖拽、分组、动态卡片、嵌套容器） | 多层级 UI 树、容器自适应、拖拽信号原生支持 | **满分**。`Control` + `Container` 体系专为"桌面级"复杂交互设计，拖拽重排、分页容器极其轻量且稳定。 | **Cocos/Unity**：UI 系统偏向游戏 HUD，做"类操作系统"的复杂交互极难实现，布局错乱和指针穿透风险高。 |
| **2. 卷尺/放大镜/化学试剂**（拖拽测量、移动镜片、试管反应） | 自定义鼠标/触摸交互、精确碰撞区域判定、实时坐标映射 | **满分**。`_input()` 和 `Control` 的鼠标过滤机制可以极低成本实现"工具拖拽到物品上触发反应"的逻辑。 | **Web + Capacitor**：DOM 虽 CSS 强，但实现"在 Canvas 上拖拽测量读数"并精确捕捉像素点，需要大量 JS 胶水代码，移动端触摸延迟风险高。 |
| **3. 知识库与侦探笔记**（碎片解锁、动态高亮、全文检索） | 富文本（RichText）渲染、BBCode/标记语言支持、文本热区点击 | **优秀**。`RichTextLabel` 原生支持 BBCode 和 meta 点击，非常适合做"线索关键词高亮与跳转"。 | **Unity**：UGUI 的 Text 富文本能力极弱，关键词高亮和自定义点击基本需要 TextMeshPro + 大量底层替换逻辑。 |
| **4. 时间线与镜头演出**（特写拉近、闪回、表情瞬切） | 场景树（SceneTree）切换、动画树（AnimationPlayer）驱动任意属性 | **满分**。`AnimationPlayer` 可控制任何节点的任何属性（包括 UI 偏移、颜色渐变），实现"特写镜头"无需挂载复杂 Cinemachine。 | **Cocos**：动画系统偏骨骼/帧动画，控制 UI 属性变化不如 Godot 直观。 |
| **5. 纯 2D 坐标逻辑**（脚印间距计算、血字离地高度换算） | 像素坐标系原生支持，无 3D 正交投影的像素偏移问题 | **满分**。Godot 是极少数真正在底层将 2D 视为一等公民的引擎，坐标直接对应像素，做"步幅与身高换算"零误差。 | **Unity**：必须使用 3D 正交相机，容易出现像素对齐（Pixel Perfect）问题，坐标换算复杂。 |

### 7.2 重审原文档中的三大"误判"

#### 7.2.1 关于"地图系统"（原评分：Web > Godot）

- **原判据**：Web 有 Leaflet.js，地图强。
- **重审结果**：**这是最大的误判。** 根据 L3（03_ 关卡设计稿 §10-§16），游戏是 **"场景节点切换"**（实验室→花园→室内→旅馆），而非"开放世界平移拖拽"。地图仅作为"证据收藏品"展示静态标点。**Godot 的 `TextureButton` + `Polygon2D` 做热点图比 Leaflet 省 100 倍性能**，且无需加载庞大 JS 库。此项 **Godot 反超**。

#### 7.2.2 关于"UI 开发效率"（原评分：Web > Godot）

- **原判据**：DOM+CSS 最灵活。
- **重审结果**：**在推理墙场景下，结论完全反转。** 推理墙需要"拖拽线索卡片到推断页签生成新结论"。DOM 拖拽需要处理复杂的 `drag/drop` 接口和回流重绘，在移动端极易卡顿。**Godot 的 `Control` 节点自带 `get_drag_data` 和 `can_drop_data`**，完美复刻操作系统级拖拽，且跨平台（PC/手机）触控反馈完全一致。此项 **Godot 碾压**。

#### 7.2.3 关于"广告 SDK"（原评分：Unity > Godot）

- **原判据**：Godot 需写 GDExtension 桥接，麻烦。
- **重审结果**：**依然如此，但不足为虑。** 根据 L1（00_ §A-2.1 和 A-2.3），**广告/内购是 v2.0+ 扩展期功能**，MVP 阶段（M1-M2）仅需单机+存档。等到 M4 需接入广告时，Godot 社区已有成熟的 `GodotAdMob` 安卓/iOS 插件，且 GDExtension 的 C++ 桥接是一次封装终生复用。此项 **不影响立项决策**。

### 7.3 基于新设计的 Godot 架构微调建议

尽管引擎选型不变，但基于《00》和《02》的最新核心循环，建议在 Godot 工程架构上做以下定向强化：

1. **明确区分"游戏世界"与"推理墙"**：
   利用 Godot 的 `CanvasLayer` 将"场景"与"推理墙 UI"彻底分离。确保无论场景如何切换（场景二到场景八），推理墙作为顶层 `Control` 永不卸载，实现"随时按出推理墙"的无缝体验。

2. **利用 `Resource` 文件模拟"知识库碎片"**：
   L3 要求的"知识碎片解锁"和"主动发现闭环"，非常适合用 Godot 的 `.tres`（自定义资源）文件作为数据载体。每个线索作为独立 `Resource`，内嵌 `unlock_conditions`（解锁条件）数组，方便策划在编辑器中直接拖拽配置，无需写 JSON。

3. **使用 `Tween` 代替 `AnimationPlayer` 做轻量交互**：
   对于"放大镜移动"、"卷尺拉出"等高频微小交互，使用 Godot 4 内置的 `Tween`（无需创建动画节点），代码控制更灵活，且能根据难度模式（简单/困难）动态调整动画速度。

### 7.4 更新后的结论

原文档的综合评分（Godot 3.75 / Cocos 3.85 / Web 3.65 / Unity 3.45）是在"通用 2D 推理游戏"框架下得出的。**在《00》《02》最新设计明确后，推理墙的 UI 复杂度、工具的精确交互需求、纯 2D 坐标计算精度，均将 Godot 从"合适"推向"唯一最优解"。**

如果重新加权（将"2D 能力"和"开发效率"的权重向 UI 交互方向倾斜），修正后的评估如下：

| 维度（修正权重） | Godot 4.x | 原评分 | 修正说明 |
|-----------------|-----------|--------|---------|
| 推理墙 UI 交互（新增，+15%） | ⭐⭐⭐⭐⭐ | — | 拖拽/分页/嵌套容器，其他引擎无法胜任 |
| 工具精确交互（新增，+10%） | ⭐⭐⭐⭐⭐ | — | `_input()` + 坐标映射，零胶水代码 |

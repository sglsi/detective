## 1. 游戏概念设计

### 1.1 游戏基本信息

| 属性 | 内容 |
|------|------|
| **游戏名称** | 《福尔摩斯探案：贝克街迷踪》 (Sherlock Holmes: Baker Street Mysteries) |
| **游戏类型** | 第一人称叙事解谜 / 互动侦探游戏 |
| **核心标签** | #侦探推理 #维多利亚时代 #分支叙事 #线索收集 |
| **故事蓝本** | 《福尔摩斯探案全集》阿瑟·柯南·道尔 |
| **主角视角** | 福尔摩斯本人（第一人称），体验大侦探的侦察思维及技能 |
| **叙事方式** | 图文并茂，场景图片 + 对话文本 + 物品观察 |

### 1.2 游戏定位与核心卖点

#### 定位声明
一款深度还原维多利亚时代伦敦氛围、以《福尔摩斯探案全集》为蓝本的互动叙事侦探游戏。玩家扮演福尔摩斯，通过场景探索、人物对话、物品观察发现线索，运用演绎推理还原犯罪真相。

#### 核心卖点 (Unique Selling Points)

| # | 卖点 | 详细描述 |
|---|------|----------|
| **USP 1** | 经典文学改编 | 忠实还原《福尔摩斯探案全集》8个经典案件（血字的研究、四签名、福尔摩斯历险记等） |
| **USP 2** | 第一人称沉浸体验 | 以福尔摩斯视角展开调查，代入感强，决策影响剧情走向 |
| **USP 3** | 推理战场系统 | 多假设并存+矛盾标记+证据关联，不是"选对答案"而是"构建推理链"；错误推理可推翻重做，零惩罚试错 |
| **USP 4** | 苏格兰场竞争暗线 | 葛莱森/雷斯垂德双警长全程翻车，侦探不孤独，你有同行衬托；7场景递进式翻车概率，增强叙事趣味 |
| **USP 5** | 三维三星评价 | 观察/推理/洞察三维独立评星（各1-3星），不扣分原则，只奖不罚；沉默线索奖励额外洞察之星 |
| **USP 6** | 维多利亚时代美学 | 精细还原19世纪伦敦的街道、建筑、服饰、氛围 |

### 1.3 市场背景与行业趋势

> **数据截止**: 2026-06-26  
> **数据来源**: Omdia 2026 全球小游戏市场预测（多家媒体转引报道，含证券时报、CNMO 科技、股海听风、搜狐等 6 月 26 日同步发布）；Niko Partners 2026 Asia+MENA 游戏市场报告作为补充

#### 1.3.1 市场体量与增速

Omdia 2026 年最新预测：全球小游戏市场规模 2026 年达 **100 亿美元**，2030 年达 **200 亿美元**，4 年实现翻倍增长，**年均复合增长率约 15%**。这一增速在全球游戏行业整体个位数增长（亚洲 + 中东 + 北非 5 年 CAGR 约 3%，Niko Partners 数据）的大背景下，尤为突出。

| 时间节点 | 全球小游戏市场规模 | 关键背景 |
|----------|------------------|----------|
| 2026 年 | 100 亿美元 | 起点规模，首次破百亿 |
| 2030 年 | 200 亿美元 | 翻倍增长，CAGR 约 15% |
| 国内 MAU（参考） | 2025 年约 6 亿，2026 预计近 7 亿 | 微信小游戏月活连续三年稳定 5 亿 |

**结论**：小游戏早已不是"小众赛道"，而是全球游戏产业最具确定性的增量方向之一。

#### 1.3.2 行业四大关键转向

Omdia 高级分析师崔辰毓指出，小游戏正经历四个关键转向：

| # | 转向 | 关键变化 | 对本项目的启示 |
|---|------|----------|----------------|
| 1 | **产品形态升级** | 从传统休闲游戏扩展至射击、动作等更具深度的品类 | 推理探案品类在休闲/深度之间，需平衡叙事深度与轻量体验 |
| 2 | **商业模式成熟** | "广告 + 应用内购（IAP）"混合变现成为主流，IAP 预计占整体收入 65%+ | 验证了"广告主导 + 内购补充"路线；需设计 IAP 锚点（线索包、徽章、章节解锁） |
| 3 | **海外市场机遇** | 海外环境改善，YouTube Instant Games、TikTok Mini Games、韩国 One Store 等渠道打开 | 维多利亚伦敦 + 柯南·道尔 IP 在海外（英语区）有天然认知度，是出海加分项 |
| 4 | **超级应用入口** | 微信/抖音/YouTube 等 Super App 仍是用户获取与留存的核心入口 | 第一阶段做网页小游戏（打开即玩），与超级应用生态天然契合 |

#### 1.3.3 与本项目的相关性

| 维度 | 本项目 | 与赛道契合度 |
|------|--------|--------------|
| **载体形式** | 网页游戏（打开即玩） | ✅ 高度契合——"Instant Games"正是小游戏赛道的核心形态 |
| **品类** | 休闲叙事 + 推理益智 | ✅ 契合"产品形态升级"——推理品类比纯休闲更有深度但门槛适中 |
| **变现** | 广告主导 + 内购补充（路线方向） | ✅ 完全契合"广告 + IAP 混合变现"主流模式 |
| **目标用户** | 文学爱好者 + 解谜玩家 + 维多利亚文化粉丝 | ⚠️ 需注意——核心用户付费意愿需探索，IAP 设计要克制 |
| **IP 资产** | 福尔摩斯 + 柯南·道尔 | ✅ 海外认知度高，是出海加分项 |
| **多端策略** | 第一阶段网页优先，第二阶段手机 App | ✅ 契合"小游戏端获客 + App 端留存"的多端并行策略 |

**核心结论**：本项目踩中小游戏赛道的主流方向，"打开即玩"载体 + 休闲益智品类 + 广告变现路线均与 Omdia 报告的关键判断一致。维多利亚时代 IP 是差异化优势，海外出海是潜在增量。

#### 1.3.4 风险与挑战

| 风险类型 | 具体表现 | 应对思路 |
|----------|----------|----------|
| **国内增速放缓** | 国内小游戏市场增速从翻倍放缓至 20-30% | 重点拓展海外增量，海外占比目标设定需前置 |
| **买量成本上行** | 2025 年在投游戏数量同比增长 54%，竞争烈度远超 App 手游 | 依赖 IP 自带流量（福尔摩斯粉丝社群）+ 内容营销，避免纯买量 |
| **海外本地化** | 不同地区用户偏好、监管政策、平台规则差异大 | 优先英语区（YouTube Playables、TikTok Mini Games），验证后再扩 |
| **IAP 转化挑战** | 核心用户多为文学爱好者，对"为虚拟道具付费"敏感 | IAP 设计需包装为"收藏价值"（徽章/典藏版）而非"通关捷径" |
| **品类竞争** | 推理赛道有《逆转裁判》《弹丸论破》《福尔摩斯：罪与罚》等成熟产品 | 强化"第一人称福尔摩斯视角"+"连锁案件伏笔"两大差异化卖点 |

### 1.5 与现有侦探推理游戏的差异化

| 维度 | 本游戏 | 《逆转裁判》 | 《弹丸论破》 | 《福尔摩斯：罪与罚》 |
|------|--------|--------------|--------------|----------------------|
| **视角** | 第一人称（福尔摩斯） | 第三人称（律师） | 第三人称（学生） | 第三人称（福尔摩斯） |
| **核心机制** | 线索发现+演绎推理 | 法庭辩论+举证 | 学级审判+投票 | 开放式调查 |
| **叙事结构** | 连锁案件（伏笔关联） | 单元剧 | 单元剧+主线 | 单元剧 |
| **难度系统** | 三档（简单/普通/困难） | 单一线性 | 单一线性 | 开放式无引导 |
| **计分系统** | 线索发现计时奖励 | 无 | 无 | 无 |
| **时代还原** | 维多利亚伦敦（重点还原） | 现代日本 | 架空未来 | 混合时代 |
| **福尔摩斯视角** | ✅ 玩家扮演福尔摩斯本人 | ❌ 不适合 | ❌ 不适合 | ❌ 玩家操控福尔摩斯 |

#### 差异化核心
1. **连锁案件伏笔系统**：不是独立案件，而是构建统一背景下的"福尔摩斯宇宙"，案件之间存在因果关联
2. **线索计分竞技性**：关键/重要/一般/其他四级线索分类，越早发现关键线索得分越高
3. **三档难度普适性**：简单引导/普通概率提示/困难无提示，覆盖休闲到核心玩家
4. **忠实原著+案件库复玩**：前期忠实原著入库，积累后可随机组合增加复玩性

**线索重要性判定规则**

#### 2.4.2 线索发现与计分机制

**时间窗口奖励公式**

```
奖励分数 = floor(基础分值 × 时间奖励系数)

时间奖励系数 = max(0, (T_max - T_current) / T_max × 0.5)

其中:
- T_max = 该线索的时间窗口上限（秒）
- T_current = 当前已用时间（秒）
- 0.5 = 最大奖励比例（50%）
- floor() = 向下取整

示例:
- 某关键线索 T_max = 300秒
- 玩家在100秒时发现: 奖励 = 100 × (200/300 × 0.5) = +33分
- 玩家在50秒时发现: 奖励 = 100 × (250/300 × 0.5) = +42分
```

**线索发现触发条件**

| 触发方式 | 描述 | 触发条件 |
|----------|------|----------|
| 对话触发 | 在NPC对话中选择特定选项 | 对话树节点 |

| 推理连接 | 将两条已知线索关联分析 | 推理笔记界面 |

#### 2.4.5 自我误导陷阱

**设计哲学：误导不是死路，是弯路**

误导线索（misleading clues）是推理游戏的必修课，但设计不好会变成"故意坑玩家"。本游戏的误导线索遵循"四级验证"原则：玩家只要交叉验证，一定能发现问题，不会卡死。

**误导线索纠错路径：**
1. 一级验证：线索本身的细节矛盾（如"血字写得很工整，但人是急死的"）
2. 二级验证：与其他线索交叉（如"情杀论"vs"财物未动"）
3. 三级验证：与假设矛盾（如"抢劫杀人"假设解释不了墙上的字）
4. 四级验证：NPC对话提示（如华生随口一句"这不太对吧"）

**《血字的研究》主要误导陷阱：**

| 编号 | 名称 | 场景 | 误导方向 | 纠错方式 | 洞察奖励 |
|------|------|------|----------|----------|----------|
| C-06 | 指甲缝白色粉末 | 场景三 | 诱导玩家怀疑"化学/医药从业者" | 与毒杀线索交叉验证后发现是普通粉笔灰 | 规避+0.3星 |
| C-18 | 刀伤角度 | 场景七 | 诱导玩家判断"左撇子凶手" | 与其他场景凶手特征（右撇子）矛盾 | 规避+0.3星 |

**重要：** 掉入误导陷阱不扣分、不惩罚，只是多走一段弯路。成功规避（即第一时间标记为"误导"）奖励洞察之星。

### 2.5 推理战场系统

> → 推理墙三层架构（展示层/逻辑层/数据层）参见《00_标准化开发目录》v2.19 B-10
> → 本章节为 M1（MVP必做）设计，M2/M3 为后续阶段扩展

#### 2.5.1 系统定位与三档分级

**定位升级：从"验证工具"到"推理战场"**

推理战场是玩家演绎推理的核心工作台——不是勾选正确答案，而是构建推理链、管理多假设、识别矛盾、逐步逼近真相的战场。错误推理不扣分、可推翻重做，只影响最终三星评价。

**三档分级路线图：**

| 档位 | 名称 | 核心机制 | 开发阶段 | 状态 |
|------|------|----------|----------|------|
| **M1** | 多假设并存 | 活跃假设列表 + 矛盾标记 + 证据关联 | MVP（v2.0） | ✅ 本版本落地 |
| **M2** | 假设竞争 | 假设评分/竞争淘汰 + 推理路径记录 + 分支回溯 | 第二阶段 | 🔲 规划中 |
| **M3** | 推理生态 | 交叉验证网络 + 演绎链可视化 + 玩家自定义推理规则 | 长期 | 🔲 远期 |

**不扣分原则（核心设计哲学）：**
- 选错假设、走错推理路径 → 不扣分、不惩罚
- 错误推理可随时推翻重做，推理链不会因错误而关闭
- 只有"发现了多少"（观察之星）、"推理质量如何"（推理之星）、"挖到多深"（洞察之星）影响评价
- 误导线索是弯路不是死路，通过交叉验证自然纠错

#### 2.5.2 M1核心机制：多假设并存

**活跃假设列表：**
- 每个场景新增 2-5 个活跃假设，跨场景累积
- 假设来源：场景探索触发/NPC对话启发/线索关联自动生成
- 玩家可随时标记"支持/反对/待验证/已排除"
- 同一场景可同时持有多个相互矛盾的假设，不强制玩家"选一个"

**假设累积示例（《血字的研究》）：**

| 场景 | 新增假设数 | 典型假设 |
|------|-----------|----------|
| 场景二（初勘现场） | 5个 | 政治谋杀/情杀/仇杀/抢劫杀人/自杀伪造 |
| 场景三（尸体检查） | 4个 | 毒杀/右撇子凶手/熟人作案/凶手穿方头靴 |
| 场景四（询问兰斯） | 3个 | 醉汉=凶手/马车夫涉案/凶手重返现场取戒指 |
| 场景五（老太婆来访） | 4个 | 同伙存在/老太婆是伪装/戒指是关键物证/凶手在追踪戒指 |
| 场景六（卡彭蒂耶） | 2个 | 卡彭蒂耶是凶手/秘书斯特兰森涉案 |
| 场景七（第二命案） | 4个 | 连环杀手/内部灭口/毒丸杀人/刀伤角度矛盾 |
| 场景八（最终对决） | 全案验证 | 10个核心假设 + 21组矛盾全解决 |

#### 2.5.3 M1核心机制：矛盾标记系统

**矛盾三类体系（21组，F/L/E 各7组）：**

| 类型 | 代码 | 定义 | 示例 |
|------|------|------|------|
| **事实矛盾** | F | 两个已知事实直接冲突 | F-03 死者无外伤 vs 现场有血迹 |
| **逻辑矛盾** | L | 推理结论与已知事实逻辑相悖 | L-02 情杀论 vs 死者随身财物未动 |
| **证据矛盾** | E | 两条证据指向不同方向 | E-01 血字"RACHE"德语 vs 凶手是美国人 |

**矛盾升级机制：**
- 跨场景累积：早期场景的矛盾在后续场景中被放大或解决
- 集中展示：推理战场中"矛盾面板"实时显示当前未解决矛盾列表
- 解决奖励：每解决一组矛盾，推理之星+0.1（累积满3星封顶）

#### 2.5.4 M1核心机制：证据关联

**关联方式：**
- 线索 ↔ 假设：每条线索可关联到一个或多个假设（支持/反对/中立）
- 线索 ↔ 线索：线索之间可建立因果关系、时间关系、空间关系
- 自动关联：系统根据规则自动建立明显关联（如同一人物的两条线索自动关联）
- 手动关联：玩家可自由建立/删除关联，体现个人推理路径

**关联可视化：**
- 推理战场以网络图形展示：节点=线索/假设，连线=关联
- 支持筛选：只看某假设的关联线索、只看未解决矛盾相关线索
- 简单模式：自动关联 + 高亮关键路径
- 困难模式：全部手动关联，系统不提示

#### 2.5.5 场景选择决策辅助

**决策辅助系统（继承自原设计，对齐不扣分原则）：**

```
玩家面临场景选择时:

✓ 推荐的场景 (有未发现线索)
  - 场景图标亮起
  - 显示"该场景有未发现线索"
  
△ 中性场景 (线索已收集完毕)
  - 场景图标灰暗
  - 显示"已探索完毕"

  - 场景图标警告色
  - 显示"进入后可能无法返回"
  
⚠ 注意：任何选择都不扣分，只影响观察之星收集完整度
```

### 2.6 三星评价体系

> → 三维独立评星机制参见《00_标准化开发目录》v2.19 B-11
> → 核心设计哲学：不扣分、只奖励，错误推理可推翻重做

#### 2.6.1 三维独立评星

游戏采用**三维三星评价体系**，三个维度独立计算、互不影响，每维各 1-3 星：

| 维度 | 名称 | 考核内容 | 3星条件 |
|------|------|----------|---------|
| ⭐⭐⭐ | **观察之星** | 场景探索完整度、线索发现数量、细节捕捉能力 | 发现 ≥90% 关键+重要线索 |
| ⭐⭐⭐ | **推理之星** | 推理战场假设质量、矛盾解决率、推理链完整度 | 解决 ≥85% 矛盾 + 核心假设全部正确 |
| ⭐⭐⭐ | **洞察之星** | 沉默线索发现、隐藏彩蛋、自我误导陷阱规避 | 发现全部沉默线索 + 规避主要误导 |

**设计原则：**
- **只奖不罚**：没有扣分机制，探索得少=星少，而非"分数低"
- **独立评价**：观察差不影响推理分，推理错不影响观察星
- **可追溯**：每颗星对应明确的达成条件，玩家知道差在哪里
- **可重玩**：单维度不满可以针对性重玩对应场景补星

#### 2.6.2 观察之星详解

**计分方式（按线索发现进度累积）：**

| 线索类型 | 权重 | 1星条件 | 2星条件 | 3星条件 |
|----------|------|---------|---------|---------|
| 关键线索 (KEY) | 高 | ≥50% | ≥75% | 100% |
| 重要线索 (IMPORTANT) | 中 | ≥30% | ≥60% | ≥90% |
| 一般线索 (NORMAL) | 低 | 不作要求 | ≥40% | ≥70% |
| 其他线索 (OTHER) | 极低 | 不作要求 | 不作要求 | ≥50% |

**难度影响：**
- 简单模式：可发现线索自动高亮，3星门槛降低（关键线索≥80%即可）
- 普通模式：轻微高亮引导，标准门槛
- 困难模式：无高亮，标准门槛

#### 2.6.3 推理之星详解

**计分方式（按推理战场表现累积）：**

| 考核项 | 权重 | 说明 |
|--------|------|------|
| 矛盾解决率 | 40% | 21组矛盾中成功解决的比例 |
| 核心假设正确率 | 30% | 10个核心假设中最终标记正确的比例 |
| 证据关联完整度 | 20% | 关键线索与假设的关联覆盖率 |
| 推理效率 | 10% | 最少推翻次数到达正确结论的效率 |

**星级门槛：**
- 1星：解决 ≥40% 矛盾 + 核心假设正确率 ≥50%
- 2星：解决 ≥65% 矛盾 + 核心假设正确率 ≥70%
- 3星：解决 ≥85% 矛盾 + 核心假设正确率 ≥90% + 关联完整度 ≥80%

**重要：** 错误假设不扣分、不惩罚。推翻重做是正常玩法，推理之星看的是"最终质量"，不是"一次做对"。

#### 2.6.4 洞察之星详解

**计分方式（额外奖励制，非主线必需）：**

| 考核项 | 单颗奖励 | 说明 |
|--------|---------|------|
| 沉默线索发现 | +0.5星/个 | D1-D4共4个，分布在4个场景（场景三/五/六/七） |
| 自我误导陷阱规避 | +0.3星/个 | C-06指甲缝白色粉末、C-18刀伤角度等 |
| 隐藏剧情触发 | +0.2星/个 | 场景细节彩蛋、NPC深度对话等 |

**设计意图：**
- 洞察之星是"额外挑战"，不是"必拿分数"
- 1星 = 发现至少1个沉默线索或2个彩蛋
- 2星 = 发现2-3个沉默线索 + 部分彩蛋
- 3星 = 发现全部4个沉默线索 + 规避主要误导 + 触发主要隐藏内容

#### 2.6.5 综合评价等级

**综合评价基于三星总数（最高9星）：**

| 评价等级 | 星级条件 | 结局表现 |
|----------|----------|----------|
| **名侦探** | 7-9星（含至少2个3星） | 获得"贝克街名侦探"称号，解锁《四签名》预告 + 开发者评论 |
| **合格侦探** | 4-6星 | 福尔摩斯认可，案件正常结案 |
| **继续推理** | ≤3星 | 提示"推理仍有疏漏"，可回溯重新推理，不强制结束 |

**与旧版对应关系：**
- 旧版"完美" ≈ 新版"名侦探"（但维度更丰富）
- 旧版"普通" ≈ 新版"合格侦探"
- 旧版"错误" → 新版"继续推理"（不是失败，只是还没做完）

### 2.8 调查进度系统

> → 三层剥离法（叙事时间/进度节点/时间锁扩展）参见《00_标准化开发目录》v2.19 B-5
> → 核心机制改名"调查进度系统"，时间仅作叙事氛围，不做硬性限制（M2+扩展）

#### 2.8.1 三层剥离法

**原时间线系统按三层职能剥离，避免"时间既是叙事元素又是机制核心"的混乱：**

| 层级 | 名称 | 职能 | MVP状态 |
|------|------|------|----------|
| **第一层** | 叙事时间 | 纯氛围元素：年月日/时辰/昼夜变化，不影响玩法 | ✅ MVP必做 |
| **第二层** | 调查进度 | 核心机制：以"已发生事件/已解锁场景"定义进度，不以"小时分钟"计算 | ✅ MVP必做 |
| **第三层** | 时间锁机制 | 扩展功能：特定事件必须在某个时间点触发，错过有替代路径 | 🔲 M2+扩展 |

#### 2.8.2 调查进度核心机制

**进度节点驱动（而非时间驱动）：**

| 规则 | 说明 |
|------|------|
| 进度推进 | 完成关键事件后推进调查进度（如"发现戒指后"解锁失物招领路线） |
| 事件驱动 | 新场景/新对话由"已完成的调查动作"触发，而非"时间到了自动触发" |
| 自由回溯 | 已探索场景可随时返回复查，不设"时间过去就回不去"的限制 |
| 非线性支持 | 部分调查路径可交换顺序，玩家选择影响推理路径但不卡死 |

**调查进度显示：**

```
┌─────────────────────────────────────────────────────────┐
│  🕐 1881年3月5日 · 上午 （仅叙事氛围）                  │
│  ━━━━━━━━●━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ │
│  承接案件  现场勘查  询问证人  ___  身份识别  ___  对决   │
│            ↑当前阶段                                     │
└─────────────────────────────────────────────────────────┘
```

**设计原则：**
- 时间只讲故事，不管玩法——玩家不用担心"花太久会出事"
- 调查进度是真·进度条——完成了什么才是关键
- 场景回溯自由——想回去再看看就回去，不设时间惩罚

#### 2.8.3 难度模式与调查进度

| 难度 | 进度显示 | 引导强度 |
|------|----------|----------|
| 简单 | 进度条高亮下一步节点 + 箭头指引 | 强引导，直接告诉你"该去哪" |
| 普通 | 进度条显示已完成/未完成节点，不标下一步 | 弱引导，靠推理判断下一步 |
| 困难 | 仅显示叙事时间，不显示进度条 | 无引导，完全自主判断 |

#### 2.8.4 时间锁机制（M2+扩展，标记待实现）

> ⚠️ 以下为 M2 扩展功能，MVP 不实现

时间锁是指"必须在某个时间段才能触发的事件"。引入原则：
- 任何时间锁必须有 ≥1 种绕过去的方式
- 任何时间锁必须有 ≥1 种补救方法
- 时间锁影响的是"路径选择"，不是"能不能通关"

典型场景（M2规划）：
- 清晨拜访证人 vs 下午拜访 → 获得不同信息（但都能推进）
- 夜间蹲守 vs 白天调查 → 不同推理路径（但殊途同归）
- 限时事件：X小时内找到某物 → 失败有替代方案，不卡关

### 2.9 苏格兰场竞争暗线

**设计定位：** 苏格兰场不是"背景板NPC"，而是全程与玩家"赛跑"的同行竞争者——双警长（葛莱森/雷斯垂德）各有一套错误推理路线，全程翻车但自我感觉良好，为严肃推理增添喜剧调味，同时反衬玩家推理的正确性。

#### 2.9.1 双警长人设与路线

| 警长 | 核心错误路线 | 性格标签 | 翻车方式 |
|------|-------------|----------|----------|
| **葛莱森** | 政治阴谋论 | 自信、浮夸、爱抢功 | 抓错人（卡彭蒂耶），自曝其短 |
| **雷斯垂德** | 情杀论 | 谨慎、死板、认死理 | 方向对了但人已死，慢一步 |

**暗线价值：**
- 叙事趣味：两个"官方侦探"全程瞎搞，增加黑色幽默
- 推理参照：玩家可以通过"警长错在哪"反推正确方向
- 节奏调节：严肃推理之间穿插喜剧桥段，张弛有度
- 成就感：看着警长们翻车，反衬玩家的正确推理

#### 2.9.2 七场景翻车路径

| 场景 | 葛莱森状态 | 雷斯垂德状态 | 关键事件 |
|------|-----------|-------------|----------|
| 场景二（初登场） | 20%自信 | 15%自信 | 各自提出初步假设，互不服气 |

#### 2.9.3 暗线触发机制

**触发方式：**
- 自动触发：进入特定场景后，双警长对话自然展开
- 可选查看：推理战场中有"苏格兰场动态"面板，实时显示双方进展
- 互动对话：玩家可选择与警长对话，套取信息或看笑话

**对主线的影响：**
- 不影响核心推理流程：暗线是"配菜"不是"主菜"
- 部分信息可利用：警长的错误假设本身就是"排除法"线索
- 评价关联：成功利用警长推理反推正确方向，推理之星额外+0.2

### 2.10 双钩子系统

**设计定位：** 场景与场景之间的自然衔接机制——不做单独的"下一场预告"UI，而是把"为什么要去下一个场景"融入场景结尾的人物对话中。每个场景结尾有两类钩子：剧情钩子（叙事驱动）+ 谜题钩子（推理驱动）。

#### 2.10.1 钩子分类

| 钩子类型 | 作用 | 示例 |
|----------|------|------|
| **剧情钩子** | 叙事驱动：人物动机/事件发展推动玩家去下一个场景 | "葛莱森说他找到新线索了，在卡彭蒂耶公寓" |
| **谜题钩子** | 推理驱动：未解之谜/新疑点激发玩家好奇心 | "戒指上刻着L·F，这是谁的名字缩写？" |

**设计原则：**
- 每个场景至少1个剧情钩子 + 1个谜题钩子
- 钩子融入对话，不做单独UI弹窗
- 钩子与下一场景直接关联，形成"提出悬念→解决悬念→新悬念"的链式结构
- 玩家可以选择跟着钩子走，也可以自由探索（调查进度系统支持回溯）

#### 2.10.2 八场景钩子示例

| 场景 | 剧情钩子 | 谜题钩子 |
|------|----------|----------|
| 场景一（贝克街实验室） | 葛莱森的电报来了，需要去现场 | "RACHE是什么意思？为什么写在墙上？" |
| 场景二（劳瑞斯顿花园街） | 尸体在室内，需要进去勘查 | "马车印是新鲜的，凶手刚离开不久——去哪了？" |
| 场景三（室内勘查） | 戒指是关键线索，需要追查来源 | "死者无外伤，怎么死的？血字谁写的？" |
| 场景四（奥德利大院） | 兰斯提到醉汉，需要确认身份 | "醉汉为什么回来？戒指是他的吗？" |
| 场景五（贝克街会客厅） | 老太婆来访取戒指，需要跟踪/设局 | "老太婆是谁？真的只是代取吗？" |
| 场景六（卡彭蒂耶公寓） | 葛莱森抓了人，对不对要验证 | "卡彭蒂耶是凶手吗？秘书斯特兰森在哪？" |
| 场景七（郝黎代旅馆） | 斯特兰森也死了，凶手还在行动 | "刀伤角度为什么奇怪？药丸从哪来的？" |

### 2.11 紧迫感机制

> → 核心触发场景：场景七（第二命案现场）
> → 设计目标：营造"慢一步就再死人"的压迫感，但不引入真实时间惩罚

#### 2.11.1 场景七核心钩子 P1-5

| 编号 | 紧迫点 | 表现方式 | 对玩家的影响 |
|------|--------|----------|-------------|
| **P1** | 霍普行踪飘忽 | NPC对话反复提及"找不到人""又消失了" | 心理压迫感：凶手在暗处，玩家在明处 |
| **P2** | 双警长双双翻车 | 葛莱森抓错人+雷斯垂德慢半拍 | 压力传递：官方靠不住，只能靠自己 |
| **P3** | 第二命案突发 | 上一个嫌疑人刚排除，又死一个人 | 节奏加速：案件在升级，不能慢慢来 |
| **P4** | 死亡方式升级 | 从毒杀→刺杀，暴力程度上升 | 危险感：凶手越来越极端 |
| **P5** | 药丸悬念 | 两粒药丸一毒一无毒，"上帝裁决" | 终极谜题：凶手的作案手法是什么？ |

#### 2.11.2 紧迫感实现方式

**纯叙事驱动，机制无惩罚：**
- 通过对话、环境描述、NPC情绪传递紧迫感
- 不设倒计时、不扣分数、不卡关
- 玩家感受到"该快点了"，但实际上慢慢玩也没问题

**表现手段：**
- 华生台词更急躁："福尔摩斯，不能再等了！"
- 场景描述更紧张："街道上警笛声此起彼伏"
- 苏格兰场动态更频繁："雷斯垂德已经出动了！"

### 2.12 竞技模式

除小说式对话推进外，游戏在关键剧情节点引入竞技场景，增加游戏趣味性和紧张感。

**竞技类型：**

| 类型 | 触发场景 | 玩法 | 失败后果 |
|------|----------|------|----------|
| 格斗 | 逮捕嫌疑人遭遇反抗 | QTE快速反应 + 方向选择 | 嫌疑人暂时逃脱，需重新追踪 |
| 对决 | 与嫌疑人正面对峙 | 推理对决——选择正确逻辑链反驳 | 嫌疑人否认指控，需补充证据 |
| 追逐 | 嫌疑人逃跑 | 路线选择 + 时机判断 | 嫌疑人逃脱，降低评分 |
| 迷宫 | 探索复杂场景（如地下通道、废弃建筑） | 方向选择 + 线索指引 | 绕路消耗时间，降低时间奖励 |

**设计原则：**
- 竞技场景是剧情的自然延伸，不是割裂的小游戏
- 失败不会导致游戏结束，但会影响评分和后续剧情走向
- 简单模式下竞技场景难度降低（如QTE按键更少、迷宫更短）
- 困难模式下竞技场景更紧张（如追逐时限更短、对决逻辑链更长）

## 3. 难度模式详细设计

### 3.1 三种模式设计思路

#### 简单模式：动态小说体验

> 偏向沉浸式侦探剧情，让玩家像读一本动态小说一样体验福尔摩斯的故事。

- **核心体验**：全程剧情完整触发，引导性推理答题，无卡点
- **推理战场**：系统自动生成假设+自动关联证据+自动标记矛盾解决
- **线索墙**：系统自动填写推论/结论，自动分类线索类型
- **侦破工具**：系统提示工具使用方向（"试试用放大镜观察血迹"）
- **调查进度**：高亮下一步节点 + 箭头指引
- **竞技场景**：难度降低，更偏向剧情体验
- **三星评价**：观察之星3星门槛降低（关键线索≥80%），推理之星自动累积
- **目标玩家**：偏好沉浸式侦探剧情的玩家，享受故事而非挑战

#### 普通模式：平衡体验

> 兼顾轻度解谜与剧情沉浸，引入推理战场的完整玩法。

- **核心体验**：推理战场M1完整开放（多假设+矛盾标记+证据关联）
- **推理战场**：假设自动列出但需玩家手动标记支持/反对；矛盾系统自动检测
- **线索墙**：概率触发时系统填写推论+分类；未触发时需玩家自己备注和分类
- **侦破工具**：概率提示工具使用；未触发时需玩家自己尝试
- **调查进度**：显示已完成/未完成节点，不标下一步
- **竞技场景**：标准难度
- **三星评价**：标准门槛，观察之星靠探索、推理之星靠推理质量
- **目标玩家**：兼顾轻度解谜 + 剧情沉浸的大众玩家

#### 困难模式：硬核侦探推理

> 完全自主推理，推理战场全手动操作，零引导零提示。

- **核心体验**：零引导硬核推理，所有判断完全由玩家做出
- **推理战场**：假设需玩家自己创建+关联+标记，系统不自动检测矛盾
- **线索墙**：只记录原始信息，推论/结论/线索类型全部由玩家自己备注和分类
- **侦破工具**：无提示，需玩家自己判断何时使用何种工具
- **调查进度**：仅显示叙事时间，不显示进度条
- **竞技场景**：更紧张（时限更短、逻辑链更长）
- **三星评价**：标准门槛但获取难度更高，挑战3星需要真正的洞察力
- **目标玩家**：硬核侦探推理爱好者

### 3.2 三种模式差异对照表

### 3.4 困难模式自探索设计

#### 困难模式核心原则

1. **零引导**: 无任何提示信息，全部由玩家自主发现
2. **隐藏关联**: 线索之间的关联不显示，需要玩家自行推理
3. **无高亮**: 所有线索外观一致，需要玩家仔细观察
4. **时间压力**: 所有线索的时间窗口缩短50%
5. **无回溯**: 已阅读对话不可重读，需玩家记录

#### 困难模式特殊机制

```
证据降解系统:
- 进入困难模式后，部分线索会随时间"降解"
- 血迹证据: 2小时后模糊 (需截图记录)
- 气味证据: 1小时后消失 (无法回溯)
- 目击证词: 可随时回看

误导线索强化:
- 困难模式中，多个推理节点存在误导性选项
- 例如：血字RACHE→选项包含"与名叫RACHEL的女人有关"（错误）
- 例如：醉汉→选项包含"与案件无关"和"凶手同伙"（错误选项）
- 玩家可能沿着错误路线走很远才发现问题
```

## 4. 叙事设计：《血字的研究》完整案例

### 4.1 故事背景与改编要点

#### 原著故事概要

《血字的研究》讲述了退役军人华生医生在伦敦与福尔摩斯合租后，接手的第一起案件：1881年，伦敦布瑞克斯顿路劳瑞斯顿花园街三号空屋中发现美国人伊诺克·J.德雷伯的尸体，死者无外伤，墙上用鲜血写着"RACHE"（德语"复仇"）。次日，其秘书约瑟夫·斯特兰森在郝黎代旅馆遇害。凶手杰弗森·霍普被捕归案，供述了十八年前的往事——露茜·费里尔被迫嫁与德雷伯，含恨而死；其父约翰·费里尔亦在逃亡中遇害。霍普为复仇追随二人至欧洲，最终以"上帝裁决"（两粒药丸一毒一无毒）的方式手刃仇人。本案揭示了正义与法律的复杂边界。

#### 改编要点

| 改编维度 | 改编策略 | 改编理由 |
|----------|----------|----------|
| **叙事视角** | 玩家扮演福尔摩斯（第一视角） | 增强代入感，体验大侦探的侦察思维 |
| **华生角色** | 学习者+引导NPC | 提供信息补充，配合观察教学 |
| **场景选择** | 线性推进+分支选择 | 忠实原著流程，关键节点提供玩家选择 |
| **难度模式** | 简单/普通/困难三档 | 简单=动态小说，普通=平衡+概率，困难=硬核推理 |
| **结局设计** | 成功破案/未成功返回继续 | 忠实原著，无多结局 |
| **隐藏线索** | 增加《四签名》伏笔 | 构建连锁案件宇宙 |

```gdscript
# 案件驱动式地图加载
var tile_map: TileMapLayer
var fog_layer: TileMapLayer

func load_case_region(case_data: CaseTileRegion) -> void:
    var bounds = case_data.get_bounding_rect()
    for row in range(bounds.position.y, bounds.end.y + 1):
        for col in range(bounds.position.x, bounds.end.x + 1):
            var tile_id = "%d%c" % [row, 64 + col]  # 如 "3D"
            _load_tile_texture(tile_id, row, col)

### 7.4 数据持久化

#### 游客模式与注册系统

```gdscript
# 身份状态
enum AuthState { GUEST, REGISTERED }
var auth_state: AuthState = AuthState.GUEST

# 游客模式：临时缓存，不持久化
func save_guest_progress() -> void:
    # 仅保存到本地临时缓存，关闭浏览器即丢失
    pass

# 保存进度时触发注册引导
func request_save() -> void:
    if auth_state == AuthState.GUEST:
        _show_register_popup()  # 弹出注册窗口
    else:
        sync_to_server()

# 注册方式
enum RegisterMode { USERNAME, PHONE, EMAIL }
func register(mode: RegisterMode, credentials: Dictionary) -> void:
    match mode:
        RegisterMode.USERNAME:
            # 用户名 + 密码，无需验证
            _register_with_username(credentials.username, credentials.password)
        RegisterMode.PHONE:
            # 手机号 + 短信验证码
            _send_sms_code(credentials.phone)
            _verify_and_register(credentials.phone, credentials.code)
        RegisterMode.EMAIL:
            # 邮箱 + 邮箱验证码
            _send_email_code(credentials.email)
            _verify_and_register(credentials.email, credentials.code)
```

## 10. AI对话系统设计

> **章节说明**：本章节定义游戏中AI生成对话系统的设计规范，采用"约束式AI生成"哲学，在保证推理逻辑确定性的同时，为每次游戏提供独特的对话体验，解决"游戏过于单一、玩一遍就没新意"的核心问题。

### 10.1 设计哲学：约束式AI生成

#### 10.1.1 核心设计理念

传统游戏对话是静态预设的——无论玩家玩多少次，NPC说的话永远相同。这导致重玩价值极低，攻略一出即索然无味。本游戏引入AI对话生成，但并非"全自由生成"，而是**"约束式生成"（Constrained AI Generation）**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    约束式AI生成模型                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────┐         ┌─────────────────┐               │
│  │   设计者固定     │         │   AI自由生成     │               │
│  │   (约束边界)     │         │   (风格表达)     │               │
│  └────────┬────────┘         └────────┬────────┘               │
│           │                            │                        │
│           ▼                            ▼                        │
│  ┌─────────────────┐         ┌─────────────────┐               │
│  │ • 必须出现的线索  │         │ • 说话语气风格   │               │
│  │ • 线索出现时机    │         │ • 闲话家常      │               │
│  │ • 禁止透露的信息  │         │ • 表达方式      │               │
│  │ • 知识边界      │         │ • 肢体语言描述   │               │
│  │ • 身份背景      │         │ • 情绪反应细节   │               │
│  └────────┬────────┘         └────────┬────────┘               │
│           │                            │                        │
│           └────────────┬────────────────┘                        │
│                        ▼                                         │
│              ┌─────────────────┐                                 │
│              │   约束式AI生成   │                                 │
│              │  确定+惊喜并存   │                                 │
│              └─────────────────┘                                 │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

#### 10.1.2 关键线索锚定机制

**什么是"锚定"（Anchoring）**

锚定是指在AI生成对话时，某些关键信息的存在与否、出现时机由设计者强制控制，AI无权绕过或更改。

| 锚定类型 | 定义 | 示例 | AI处理方式 |
|----------|------|------|------------|
| **存在锚** | 某线索必须存在 | 血字"RACHE" | AI不可否认其存在 |
| **时机锚** | 某线索何时可透露 | 玩家出示证据后才谈戒指 | AI不可提前说出 |
| **边界锚** | NPC知识的范围 | 卡彭蒂耶太太不知道凶手是谁 | AI不可让夫人说出凶手 |
| **禁止锚** | 绝对禁止透露 | 困难模式下禁止提示 | AI绝对不可提示 |

**锚定实现流程**

```
场景: 卡彭蒂耶太太的公寓

[玩家进入场景]

设计锚定:
├── 必须存在: 夫人手上的戒指（外观必须与死者戒指相似）
├── 时机锚: 玩家出示死者戒指前，夫人不会主动提起自己戒指
├── 边界锚: 夫人不知道"杰弗森·霍普"这个名字
└── 禁止锚(困难): 夫人不会说"也许该看看我的戒指"

[AI生成逻辑]

AI被允许做的事:
├── 用不同方式描述夫人的紧张神态
├── 聊家常、天气、伦敦雾霾
├── 对"您丈夫昨晚在哪"这类问题给出口音独特的回答
└── 用维多利亚时代的方式表达恐惧

AI被禁止做的事:
├── 主动说"您看我手上的戒指"
├── 透露凶手姓名
├── 在玩家未出示证据前就承认认识死者
└── 提供任何推理提示（困难模式）
```

#### 10.1.3 对话风格自由机制

在锚定边界内，AI可以自由发挥的部分：

| 自由度维度 | 约束边界 | 自由发挥空间 |
|------------|----------|--------------|
| **语气风格** | 必须符合角色身份 | 可以活泼/阴沉/刻薄/热情 |
| **表达方式** | 不能说谎/透露禁止信息 | 用什么词、什么比喻、什么典故 |
| **闲话家常** | 不影响主线 | 聊天气、聊邻居、聊今天早餐 |
| **肢体语言** | 不能与情绪状态矛盾 | 具体怎么表现紧张（搓手/绞手指/目光游移） |
| **对话长度** | 根据场景类型 | 可以简短敷衍或长篇大论 |

#### 10.1.4 困难模式下的误导AI生成

困难模式中，AI可主动生成**合理的误导信息**，增加挑战难度：

```
困难模式误导示例:

NPC: 邮差 (误导向: "我没看到可疑的人")
├── AI生成: 邮差会说"我一整天都在送信，确实看到一个穿黑衣的高个子，
│          但那有什么奇怪的？伦敦满大街都是穿黑衣的。"
├── 实际影响: 误导玩家去追查"黑衣人"，而非真正的凶手特征
└── 设计目的: 测试玩家是否会轻信表面证词
```

### 10.2 NPC行为约束系统

#### 10.2.1 三层属性架构

每个NPC由三层独立的属性构成，层与层之间有明确的交互规则：

```
┌─────────────────────────────────────────────────────────────────┐
│                      NPC属性三层架构                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    第一层: 身份层 (IDENTITY)            │   │
│  │                         【固定·不可改写】                  │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │  • 社会身份: 卡彭蒂耶太太 / 警探 / 邮差                    │   │
│  │  • 职业背景: 摩门教徒 / 伦敦警察 / 服务业                │   │
│  │  • 与案件关系: 受害者妻子 / 报案人 / 证人                │   │
│  │  • 维多利亚时代背景: 阶级、口音、礼仪规范                 │   │
│  │                                                          │   │
│  │  ⚠️ AI绝对不可改写身份层，只能在此边界内生成对话          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                   │
│                              ▼                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    第二层: 知识层 (KNOWLEDGE)            │   │
│  │                         【半固定·由锚定控制】              │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │  • 必须透露 (MUST_REVEAL): 该NPC主动必须说出的信息       │   │
│  │    例: 卡彭蒂耶太太"我丈夫昨晚没回家"                      │   │
│  │                                                          │   │
│  │  • 可以透露 (CAN_REVEAL): 有条件时可以透露的信息          │   │
│  │    例: 出示戒指后"是的，我也有一枚相似的..."              │   │
│  │                                                          │   │
│  │  • 禁止透露 (NEVER_REVEAL): 绝对不能透露的信息           │   │
│  │    例: 夫人不知道凶手身份                                │   │
│  │                                                          │   │
│  │  • 无知区域 (UNKNOWN): 该NPC真的不知道的事              │   │
│  │    例: 夫人不知道摩门教长老的秘密                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              │                                   │
│                              ▼                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    第三层: 性格层 (PERSONALITY)          │   │
│  │                         【AI自由发挥】                    │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │  • 说话风格: 正式 / 随意 / 紧张 / 傲慢                    │   │
│  │  • 口头禅: 特定词汇或短语（可随机选择）                   │   │
│  │  • 情绪反应: 对特定话题的典型反应模式                     │   │

#### 10.2.3 线索泄露规则

**线索解锁条件系统**

每条线索都有明确的"解锁条件"，只有满足条件，AI才会生成包含该线索的对话：

| 线索ID | 线索内容 | 解锁条件类型 | 解锁条件 | 提示模式 |
|--------|----------|--------------|----------|----------|
| clue_004 | 卡彭蒂耶太太的证词 | evidence_required | 玩家出示礼帽 | 承认德雷伯是房客 |
| clue_006 | 戒指的秘密 | item_observed | 玩家已观察过死者戒指 | 直接承认 |
| clue_005 | 马车车轮印分析 | dialogue_progress | 与莱斯特雷德对话超过3轮 | 自然透露 |
| clue_007 | 摩门教背景 | topic_mentioned | 玩家主动询问宗教话题 | 部分透露 |

**解锁条件类型定义**

```
UNLOCK_CONDITIONS:
│
├── 1. evidence_required
│   ├── 描述: 需要玩家出示特定物品证据
│   ├── 示例: "出示死者的戒指" → 夫人承认自己也有类似戒指
│   └── 触发: 对话选项"出示证据"
│
├── 2. item_observed  
│   ├── 描述: 需要玩家已观察过某物品
│   ├── 示例: "玩家已观察死者戒指" → 夫人可能被打断借口
│   └── 触发: 物品观察记录系统
│
├── 3. dialogue_progress
│   ├── 描述: 需要与NPC进行足够轮次对话
│   ├── 示例: "对话超过3轮" → NPC放松警惕透露更多信息
│   └── 触发: 对话轮次计数器
│
├── 4. topic_mentioned
│   ├── 描述: 需要玩家主动提及特定话题
│   ├── 示例: "玩家询问宗教相关" → NPC展开摩门教话题
│   └── 触发: 玩家对话关键词检测
│
├── 5. clue_combination
│   ├── 描述: 需要已发现特定线索组合
│   ├── 示例: "发现血字+摩门教背景" → NPC承认知道更多
│   └── 触发: 线索发现状态追踪
│
└── 6. time_elapsed
    ├── 描述: 需要游戏时间超过阈值
    ├── 示例: "游戏时间超过20分钟" → NPC可能疲惫透露更多
    └── 触发: 游戏内计时器
```

### 10.3 NPC情绪状态系统

#### 10.3.1 情绪参数定义

每个NPC有四个核心情绪参数，每个参数范围0-100：

| 参数 | 定义 | 含义 | 影响 |
|------|------|------|------|
| **紧张度 (Tension)** | NPC当前的紧张/焦虑程度 | 越高=越紧张 | 说话结巴、矛盾多、回避问题 |
| **信任度 (Trust)** | NPC对玩家的信任程度 | 越高=越信任 | 主动提供信息、减少隐瞒 |
| **隐藏度 (Concealment)** | NPC主动隐藏信息的程度 | 越高=越隐藏 | 回避、撒谎、转移话题 |
| **情绪稳定性 (Stability)** | NPC情绪的稳定程度 | 越低=越不稳定 | 容易被激怒/崩溃 |

#### 10.3.2 初始情绪状态设定

根据NPC角色特点设定初始情绪值：

| NPC | 紧张度 | 信任度 | 隐藏度 | 情绪稳定性 | 设定理由 |
|-----|--------|--------|--------|------------|----------|
| 卡彭蒂耶太太 | 50 | 40 | 40 | 70 | 保护女儿，对德雷伯行为愤怒 |
| 莱斯特雷德 | 40 | 50 | 20 | 80 | 专业探员，信任福尔摩斯 |
| 华生 | 20 | 90 | 5 | 90 | 好友，完全信任 |
| 邮差(证人) | 50 | 40 | 30 | 70 | 普通证人，略紧张 |
| 克莱蒂格酒店老板 | 30 | 30 | 50 | 75 | 生意人，有所保留 |

#### 10.3.3 玩家行为对情绪的影响

```
┌─────────────────────────────────────────────────────────────────┐
│                    情绪影响矩阵                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  玩家行为                    →    情绪变化                       │
│  ─────────────────────────────────────────────────────────────  │
│                                                                  │
│  [直接质问]                 →    紧张度+15, 隐藏度+10            │
│  [出示关键证据]             →    紧张度+10, 隐藏度-20            │
│  [态度友好/表示理解]         →    信任度+15, 紧张度-10            │
│  [态度强硬/威胁]            →    紧张度+20, 隐藏度+15, 信任度-10 │
│  [提及相关话题]             →    隐藏度+5 (试探性)               │
│  [放过当前话题转聊别的]      →    紧张度-10, 隐藏度-5             │
│  [华生在旁帮助]             →    信任度+10                       │
│  [等待沉默]                 →    紧张度+5/轮                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

情绪影响示例 - 卡彭蒂耶太太对话流程:

[初始状态]
紧张度:50 | 信任度:40 | 隐藏度:40 | 稳定性:70

[玩家问: "德雷伯先生是怎样的房客?"]
→ 紧张度: 50+5=55
→ AI反应: "他...他是个规矩的租客，大多时候。"

[玩家态度选择: 友好+表示理解]
→ 信任度: 40+15=55
→ 紧张度: 55-10=45
→ AI反应: "谢谢您的理解，先生。其实我...我有些话想说。"

[玩家出示礼帽]
→ 紧张度: 45+10=55
→ 隐藏度: 40-20=20
→ AI反应: "这顶帽子...是德雷伯先生的，我认得。"
   (不再否认，开始动摇)

[玩家继续追问]
→ 紧张度: 75+15=90 (接近崩溃边缘)
→ 情绪稳定性: 60-15=45
→ AI反应: "我说...我说就是了...是的，我认识他..."
```

#### 10.3.4 AI根据情绪参数调整对话风格

AI根据当前情绪参数动态调整输出风格：

**紧张度影响 (Tension: 0-100)**

| 紧张度区间 | 对话表现 | 示例文本 |
|------------|----------|----------|
| 0-30 (平静) | 说话流畅，逻辑清晰 | "是的，那天晚上我确实看到他出门了。" |
| 31-50 (轻微) | 偶有停顿 | "那晚...对，他大概七点左右出去的。" |
| 51-70 (紧张) | 说话结巴，重复 | "他、他大概...是晚上吧...七点左右..." |
| 71-85 (高度) | 语速加快，矛盾 | "我没说谎！我就是看到他出门的！等等，我说的是看见..." |
| 86-100 (崩溃) | 情绪失控，可能透露 | "够了！你们为什么总是不相信我！我...我其实..." |

**信任度影响 (Trust: 0-100)**

| 信任度区间 | 对话表现 | 示例文本 |
|------------|----------|----------|
| 0-30 (戒备) | 只回答问题，不多说 | "我不确定这与案件有什么关系。" |
| 31-50 (观望) | 有限度配合 | "我可以告诉你这些，但仅此而已。" |
| 51-70 (配合) | 适度主动 | "既然您问了，我想起那天还有件事..." |
| 71-90 (信任) | 主动提供信息 | "福尔摩斯先生，我还有些事想告诉您..." |
| 91-100 (完全) | 知无不言 | "我完全信任您的判断，我把知道的都告诉您。" |

**隐藏度影响 (Concealment: 0-100)**

| 隐藏度区间 | 对话表现 | 示例文本 |
|------------|----------|----------|
| 0-30 (坦诚) | 直接回答，不回避 | "是的，我确实认识他。" |
| 31-50 (谨慎) | 模糊回答 | "我与他只是点头之交。" |
| 51-70 (隐瞒) | 转移话题/部分承认 | "那是很久以前的事了。您今天来是为了什么？" |
| 71-85 (掩饰) | 明显撒谎 | "我从未见过这个人，更别说戒指了。" |
| 86-100 (深藏) | 完全否认+反问 | "您在说什么？我完全听不懂。您有搜查令吗？" |

#### 10.3.5 玩家通过情绪变化判断NPC是否说谎

情绪状态为玩家提供重要的"元推理"信息：

```
玩家判断系统使用指南:

│                                                                  │

简单模式辅助:

### 10.5 难度模式与AI对话的配合

#### 10.5.1 三种难度的AI对话差异

| 对话维度 | 简单模式 (Easy) | 普通模式 (Normal) | 困难模式 (Hard) |
|----------|-----------------|-------------------|------------------|
| **主动提示** | 频繁自然提示 | 偶尔暗示(25%概率) | 无任何提示 |
| **线索透露** | 直接承认，快速推进 | 有条件承认，需引导 | 尽可能否认/回避 |
| **误导信息** | 无 | 偶尔(10%概率) | 频繁合理误导 |
| **信任门槛** | 降低30% | 标准 | 提高20% |
| **紧张度衰减** | 快(友好即降) | 正常 | 慢(需多次互动) |
| **情绪反馈** | 详细显示 | 中等显示 | 隐藏/模糊显示 |
| **对话长度** | 偏长(详细解释) | 适中 | 偏短(惜字如金) |

### 10.7 一致性保障机制

#### 10.7.1 事实一致性检查 (Fact Consistency Check)

**问题**：AI可能在对话中产生与已知事实矛盾的信息。

**解决方案**：建立事实数据库，AI生成对话前必须验证。

```
┌─────────────────────────────────────────────────────────────────┐
│                    事实一致性检查流程                            │
├─────────────────────────────────────────────────────────────────┤

│  1. [生成前检查]                                                  │
│     AI在生成对话前，读取当前上下文:                              │
│     ├── 已确认的事实列表                                         │
│     ├── 玩家已发现的线索                                         │
│     ├── NPC当前的情绪状态                                        │
│     └── 之前的对话历史摘要                                       │
│                                                                  │
│  2. [约束注入]                                                    │
│     将以下约束注入Prompt:                                        │
│     "你必须遵守以下事实（不可矛盾）:                             │
│      • 德雷伯已死亡(非失踪)                              │
│      • 夫人声称丈夫失踪(但实际是她报案)                          │

│      • ..."                                                       │

#### 10.7.2 线索计数系统 (Clue Tracking System)

**问题**：关键线索可能出现得太早或太晚，影响游戏体验。

**解决方案**：实时追踪已揭露线索数量，控制AI对话中的信息透露节奏。

```

├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  阶段1: 探索期 (已发现线索 < 30%)                                 │
│  ├── AI表现: 观望、防守、不透露核心信息                          │
│  ├── NPC倾向: 讲述背景而非关键事实                               │
│  └── 举例: "是的，德雷伯先生最近确实常来..."                  │
│                                                                  │
│  阶段2: 调查期 (已发现线索 30%-60%)                               │
│  ├── AI表现: 适度配合、开始透露信息                              │
│  ├── NPC倾向: 回答具体问题，但不完全坦白                         │
│  └── 举例: "那天他来的时候，神色慌张，我还以为..."                 │
│                                                                  │
│  阶段3: 深入期 (已发现线索 60%-80%)                               │
│  ├── AI表现: 明显动摇、可能主动提供信息                          │
│  ├── NPC倾向: 不再隐瞒，渴望倾诉或进一步确认                      │
│  └── 举例: "既然您已经知道这么多...是的，那枚戒指..."              │
│                                                                  │
│  阶段4: 收束期 (已发现线索 > 80%)                                 │
│  ├── AI表现: 全力配合、可能完全坦白                              │
│  ├── NPC倾向: 承认一切，解释动机                                  │
│  └── 举例: "事已至此，我把一切都告诉您..."                        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

AI生成时注入的阶段约束:

- 30-60%: Provide some detail when asked directly
- 60-80%: Begin to open up, volunteer small information
- Over 80%: Ready to confess if confronted with evidence"
```

    "reference_past": "AI可以提及之前的对话内容",
    "emotion_continuity": "情绪状态必须从前一次对话延续",
    "contradiction_flag": "检测到矛盾时记录并修复"
  }
}

AI生成时参考历史:

"Previous dialogue with Mrs. Charpentier:
- Turn 1: She mentioned Drebber was a tenant who behaved oddly
- Turn 2: She opened up slightly when treated kindly, mentioned Mr. Drebber
- Turn 3: When shown the hat, she identified it as Drebber's

Current situation: Player asks about the ring again

AI must:
✓ Reference her previous admission
✓ Not deny knowing about the ring
✓ Show increased tension (she's been caught)
✗ Cannot now claim she doesn't know Drebber"
```

#### 10.7.4 上下文记忆管理 (Context Memory Management)

对于跨场景的长期对话（如华生全程陪同），需要管理更长的记忆：

```
长期记忆 vs 短期记忆:

┌─────────────────────────────────────────────────────────────────┐
│                    记忆层次结构                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  [长期记忆 - 永久保留]                                            │
│  ├── NPC身份背景                                                 │
│  ├── 已确认案件事实                                               │
│  ├── 玩家角色(福尔摩斯/华生)性格特征                             │
│  └── NPC与玩家的历史关系                                         │
│                                                                  │
│  [中期记忆 - 当前场景保留]                                        │
│  ├── 当前场景的对话历史(最近10轮)                                 │
│  ├── 本场景的情绪变化轨迹                                         │
│  └── 本场景已透露的线索列表                                       │
│                                                                  │
│  [短期记忆 - 当前对话轮次]                                         │
│  ├── 本次对话的上下文                                             │
│  ├── 刚才讨论的话题                                               │
│  └── 刚做出的反应                                                 │
│                                                                  │
│  记忆调用规则:                                                    │
│  ├── 每次生成对话前，读取【中期记忆】作为主要上下文               │
│  ├── 引用过去事件时，从【长期记忆】提取                           │
│  └── 保持【短期记忆】简洁，只记录关键转折点                       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

记忆容量限制 (防止上下文溢出):

最大记忆条目: 50条对话记录
超出时: 自动压缩，保留关键信息摘要
  原: "Turn 5: Mrs. Charpentier mentioned seeing a tall man"
  压缩: "提及曾见高个子男人"
```

#### 10.7.5 异常处理机制

```python
# 一致性异常处理伪代码

class ConsistencyChecker:
    def __init__(self):
        self.fact_db = FactDatabase()
        self.clue_tracker = ClueTracker()
        self.dialogue_log = DialogueLog()
    
    def validate_ai_output(self, output, context):
        """验证AI输出的一致性"""
        issues = []
        
        # 1. 事实一致性检查
        for fact in self.fact_db.get_applicable_facts():
            if self._contradicts_fact(output, fact):
                issues.append(f"Fact contradiction: {fact.id}")
        
        # 2. 锚定规则检查
        for anchor in context.get_active_anchors():
            if not self._follows_anchor(output, anchor):
                issues.append(f"Anchor violation: {anchor.id}")
        
        # 3. 历史一致性检查
        history = self.dialogue_log.get_recent_history(context.npc_id)
        if self._contradicts_history(output, history):
            issues.append("Dialogue history contradiction")
        
        # 4. 情绪连续性检查
        if not self._matches_emotion(output, context.current_emotion):
            issues.append("Emotion continuity issue")
        
        return issues
    
    def handle_violation(self, issue, output):
        """处理违规情况"""
        if issue.severity == "critical":
            # 拒绝输出，重新生成
            return {"action": "regenerate", "reason": issue.description}
        elif issue.severity == "warning":
            # 尝试修正
            return {"action": "patch", "patch": self._generate_patch(issue)}
        else:
            # 记录但放行
            return {"action": "log_and_continue"}
```

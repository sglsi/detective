 # 维多利亚伦敦探案项目 - 核心设计思路
 
 ## 项目概述
 
 以《福尔摩斯探案全集》为蓝本，设计一款应用类推理小游戏。玩家以福尔摩斯的第一视角，通过对线索的观察、与剧中人物的对话，一步一步揭开犯罪真相。虽是游戏，也是一本动态的小说——玩家在游戏中带入侦探过程或现场，就像看一部电影，达到身临其境的感觉。
 
 ---
 
 ## 一、核心理念
 
 ### 1.1 游戏本质
 
 > 这不只是一块游戏，更是一本动态的小说、一部可以参与的电影。
 
 - **游戏 = 动态小说**：玩家不是在"玩关卡"，而是在"经历故事"
 - **身临其境**：以福尔摩斯视角展开调查，代入侦探思维与技能
 - **忠实原著**：以《福尔摩斯探案全集》的固定情节及线索细节为主，经得起原著党的推敲
 
 ### 1.2 核心差异化优势
 
 | 优势 | 说明 |
 |------|------|
 | **经典文学改编** | 忠实还原《福尔摩斯探案全集》，以原著为蓝本而非架空 |
 | **第一视角沉浸** | 玩家扮演福尔摩斯本人，体验大侦探的侦察思维及侦察技能 |
 | **真实历史地图** | 基于1864年斯坦福图书馆版伦敦真实地图，真实数据、真实场景 |
 | **连锁案件叙事** | 案件间存在隐藏关联，一个大故事背景贯穿系列案件 |
 | **案件库复玩** | 积累的案件可随机组合，增加游戏复玩性 |
 
 ### 1.3 地图哲学
 
 > 地图不是一个"工具"，而是一个"收藏品"。
 
 地图基于1860年代伦敦真实地图——真实的数据，真实的场景，经得起推敲。玩家如果想在游戏中探索当年历史上著名的场景，可以在游戏中去发掘。但地图不去展示所有细节，只展示和案件相关的场景、历史上著名的地点与建筑，就像1851年Tallis & Rapkin万国博览会纪念版伦敦地图中四周环绕着伦敦标志性建筑的精美插画，让玩家有代入感或探索欲。
 
 - 每个案件只展示与其相关部分的地图，不覆盖整个伦敦
 - 随着案件不断探索，逐步展开伦敦的整个地图
 - 与案件相关的场景如果是历史上著名的景点，用真实历史图片，增加真实感
 - 随着后续游戏开发，不断在地图中增加新的与案件相关的准确位置信息，不断充实与完善

 ## 二、游戏核心玩法
 
 ### 2.1 双线叙事驱动
 
 场景中展现故事情节主要通过两个方面：
 
 **1）人物对话（主线驱动）**
 - 对话是展开案件故事情节的主要方法，像福尔摩斯的原著小说一样，推动故事情节不断向前发展
 - 人物对话过程中，可结合人物表情、势态来反映人物心理或故事情节，增加游戏趣味性
 
 **2）场景观察（辅助推理）**
 - 可以像福尔摩斯一样，通过对场景中相关线索进行"鬼才"般的洞察，发现其中有用的价值
 - 例如：对不同脚印的分析，发现哪些是凶手的、哪些是受害者的、哪些是其他人员的，从而推断凶手的主要特征
 - 场景主要通过动态图片方式呈现，具体细节可通过点击放大或弹出细节图片呈现
 
 ### 2.2 线索发现与分类
 
 玩家通过对场景中相关人物的对话，以及场景中相关物品的观察及分析，发现案件相关线索。
 
 **线索等级分类：**
 
 | 线索类型 | 说明 | 得分权重 |
 |----------|------|----------|
 | 关键线索 | 直接指向案件核心真相 | 10 |
 | 重要线索 | 推动案情重大进展 | 5 |
 | 一般线索 | 补充案情细节 | 2 |
 | 其他线索 | 背景信息、环境细节 | 0 |
 | 误导线索 | 故意误导玩家的线索 | 0（不扣分） |
 
 **推理墙：**
 
 推理墙采用**三层架构**：核心基础层（开局常驻）+ 扩展进阶层（分阶段解锁）+ 全局支撑层。
 
 **核心基础层（开局常驻）：**
 - **线索库**：全案证据统一收纳中心，线索4种状态（未发现/已收集/已验证/已失效），状态变更联动所有关联模块
 - **推理看板**：核心推理操作区域，采用**假设三级树状结构**（核心问题→主假设→子假设），支持双交互模式
   - 拖拽模式（默认）：拖拽线索卡片至假设节点，操作路径短、学习成本低
   - 自由连线模式（进阶可选）：自由连接任意节点，支持线索→假设/线索→线索/假设→假设三种连线类型
 - **四级验证体系**：假设验证等级分4档（矛盾冲突0-30%/证据不足31-60%/倾向成立61-85%/已获证实86-100%），每档对应明确的玩家行动指引
 - **结论里程碑**：案件侦破进度可视化，随假设「已获证实」逐步点亮事实节点
 
 **扩展进阶层（分阶段解锁）：**
 - **可信度评估**：第一章现场勘查后解锁，按线索属性+场景条件计算可信度，前台展示高/中/低三档
 - **矛盾疑点册**：线索对比台（全难度常驻）+ 福尔摩斯检验（主动扫描）+ 被动矛盾提示（分难度配置）
 - **案件时间轴**：第三章多场景调查时解锁，事件驱动时序推进，时效差异影响线索信息完整度
 
 **全局支撑层：**
 - 评价体系、难度适配系统、存档系统、数据统计
 
 **玩家认知系统：**
 - **线索标签体系**：所有线索统一配置标准化标签，用于自动匹配假设、计算可信度、识别矛盾
   - 内容标签：标识线索核心信息（如「车轮印」「身高特征」「毒药」「证词」）
   - 属性标签：标识线索来源类型（如「直接物证」「目击证词」「二手传闻」「嫌疑人陈述」），用于可信度计算
   - 关联标签：标识线索所属推理方向（如「马车相关」「凶手特征」「动机相关」），用于自动匹配假设节点
 - **支持度计算**：单条线索贡献值 = 线索可信度 × 标签匹配系数（核心标签完全匹配×1.0/强相关×0.7/弱相关×0.3/无匹配×0.0），假设总支持度 = 所有支持线索贡献值之和 - 所有反对线索贡献值之和
 - **核心玩法循环**：探索收集→关联推理→矛盾排查→验证判断→事实确认→闭环结案
 - 按照玩家对案件的理解形成线索认知，依次进入不同场景调查，在调查中不断调整思路
 - 越能尽早发现关键线索、越能准确建立线索与假设关联的，得分越高
 
 ### 2.3 三种难度模式
 
 | 机制模块 | 简单/助手模式 | 普通模式 | 困难模式 |
 |----------|---------------|----------|----------|
 | **场景线索提示** | 高亮标记所有可交互点 | 微光标记关键交互点 | 无任何提示标记 |
 | **默认交互模式** | 拖拽模式，支持手动切换 | 拖拽模式，支持手动切换 | 拖拽模式，支持手动切换 |
 | **可信度展示** | 显示等级+详细来源说明 | 显示高/中/低三档等级 | 不显示可信度标识，玩家自行判断 |
 | **假设引导** | 自动高亮优先验证的假设方向 | 弱提示推荐调查方向 | 无引导，完全自主判断 |
 | **矛盾被动提示** | 默认开启，新矛盾自动推送 | 默认关闭，支持手动开启 | 强制关闭，不可手动开启 |
 | **福尔摩斯检验** | 无使用次数限制 | 单场景限用2次 | 单案件限用3次 |
 | **验证报告内容** | 完整展示支持/反对证据+明确行动建议 | 展示核心依据+模糊方向建议 | 仅显示验证等级，无详细分析 |
 | **时间时效规则** | 无线索失效，所有线索永久有效 | 线索存在信息完整度差异，不会完全失效 | 严格时序规则，线索可完全失效 |
 | **凶手行动线** | 无 | 无 | 同步运行，影响结局走向 |
 
 **普通模式的概率调整机制：**
 - 系统检测玩家进度情况
 - 根据进度情况调整提示概率（进度落后则提高概率，进度正常则降低概率）
 - 概率形式增加复玩性——玩家复玩时可获得不一样的游戏过程和感觉
 - 同时避免困难模式的无提示导致玩家茫然——如果没有适当的提示，玩家可能不知道应该干什么、能干什么，从而对游戏失去兴趣

 ## 三、侦破工具与时间线
 
 ### 3.1 侦破工具系统
 
 系统提供一套基础侦破工具，玩家可借助这些工具进行测量、抓捕等操作：
 - **放大镜**：细节放大观察，发现肉眼不可见的痕迹
 - **卷尺**：精确测量距离、脚印大小等
 - **黄页**：查询伦敦地址、机构、人物信息
 - **手铐**：逮捕嫌疑人
 - **绳索**：捆绑、攀爬、标记
 - **化学试剂盒**：简易化学检验
 - **石膏粉**：提取脚印/工具痕迹
 
 工具栏常驻界面底部，工具与场景物品组合可触发新发现。简单模式下系统提示工具使用方向，困难模式下无提示。
 
 ### 3.2 时间线系统（事件驱动）
 
 游戏采用**事件驱动**的时间推进机制，而非固定时间计量：
 - **事件驱动**：时间只在玩家做出推进性动作后才推进，以"剧情事件"作为时间单位
 - **事件类型**：触发型（到达地点/见NPC）、对话型（关键选项）、调查型（完成系统性调查）、等待型（NPC外出）、跨日型（昼夜交替）
 - **时间不可逆（默认规则）**：已过去的时间不可回溯，玩家的选择有不可逆性
 
 **时间锁机制：**
 - **设计定位**：时间锁是"软锁而非硬锁"——让玩家感受到调查有时间压力，但不会因错过窗口而陷入死局
 - **核心原则**：**任何时间锁都至少有1种"绕过去"方法和1种"补救"方法**
 - **绕过去方法**：替换路径（并行调查线）、换NPC（间接知情者）、时间跳转（跳到下一时间窗）、放下案件（暂时离开保留进度）、求助华生
 - **补救方法**：事后补回（回到关键地点补做）、补救事件（触发支线还原信息）、NPC主动告知、推理墙回顾（福尔摩斯引用未发现线索）、二周目解锁
 - **时间锁字段**：`time_lock`（是否锁住）、`bypass_methods`（绕过去方法列表）、`remedy_methods`（补救方法列表）
 
 ### 3.3 竞技模式
 
 除小说式对话推进外，游戏在关键剧情节点引入竞技场景：
 - **格斗**：逮捕嫌疑人遭遇反抗 → QTE快速反应 + 方向选择
 - **对决**：与嫌疑人正面对峙 → 推理对决，选择正确逻辑链反驳
 - **追逐**：嫌疑人逃跑 → 路线选择 + 时机判断
 - **迷宫**：探索复杂场景 → 方向选择 + 线索指引
 
 竞技场景是剧情的自然延伸，失败不会导致游戏结束，但会影响评分和后续剧情走向。

 ## 四、连锁案件架构
 
 ### 4.1 统一故事背景
 
 > 游戏设计为在一个大的故事背景下展开，不是《福尔摩斯探案全集》中一个一个相对独立的故事。
 
 - 福尔摩斯通过侦查破获当前案件、抓住嫌疑人
 - 同时在侦破过程中，发现隐藏的剧情，为下一个案件埋下伏笔
 - 案件之间顺滑转换，增加玩家的探索欲
 
 **示例——《血字的研究》→《四签名》：**
 - 《血字的研究》中的犯罪嫌疑人杰弗森·霍普，可能与《四签名》中的某人有联系
 - 此人也许给霍普提供了被害人伊诺克·德雷伯的踪迹
 - 也许提供钱财以方便霍普在伦敦生活并开展他的"正义"行动
 - 根源可能是此人想侵占德雷伯在伦敦的商业及财产，或存在商业冲突/竞争
 - 从而为下一个案件埋下伏笔，顺滑转换到《四签名》
 
 **更深层架构：**
 - 《四签名》结案后，玩家发现嫌疑人背后更深有一个更大的组织
 - 《福尔摩斯探案全集》中的大部分案件都有可能处于这个统一的背景或线索下展开
 
 ### 4.2 案件库与复玩性
 
 **第一阶段——忠实原著入库：**
 - 建立案件库，将前期福尔摩斯游戏中的不同案件分别设置于案件库中
 - 福尔摩斯的相关游戏按照忠实于原著放入库中
 - 不影响原著剧情
 
 **第二阶段——案件随机组合：**
 - 待相关案件积累到一定程度后，后续其它系列可从案件库中随机提取
 - 在不影响游戏剧情的情况下，按合理度配置正确的线索细节
 - 可调整元素：案件背景时间、地点、人物身高体重等信息，在不影响整体案情的情况下适度调整
 - 后续故事情节的发展按这些案件及细节进行演变
 - 增加游戏的复玩性

 ## 五、地图系统设计
 
 ### 5.1 案件驱动式地图展开
 
 - 不预加载完整80格地图，每个案件只加载与该案件关联的瓦片区域
 - 以《血字的研究》为例：关联瓦片为3D、3E、4E、5F、5H、6F、6G、6H、7F、7G，包围矩形为行3-7、列D-H共25格
 - 下一个案件在当前基础上扩展其关联瓦片，可能重叠或新增区域
 - 已探索区域永久保留，迷雾只覆盖未到达的区域
 
 ### 5.2 地图内容原则
 
 - 只展示和案件相关的场景、历史上著名的地点/建筑
 - 著名景点用真实历史图片，增加真实感
 - 随案件进展不断补充新的准确位置信息
 - 不断充实与完善地图，让地图成为收藏品
 
 ### 5.3 界面视觉冲击力
 
 > 玩家打开游戏的第一眼，就应该被震撼——像超级大片的宣传海报，图像、文字、色彩、布局共同营造神秘感与震撼力。
 
 - **图像**：大尺度、高对比度的角色剪影或场景主视觉
 - **文字**：维多利亚复古衬线字体，标题字母间距加大，配合做旧纹理
 - **色彩**：暖棕底色 + 金色强调（煤气灯光）+ 深褐暗部（迷雾）
 - **布局**：戏剧化构图，角色偏侧/逆光，留白处弥漫雾气
 - **动效**：标题渐显、雾气流动、煤气灯闪烁

 ## 六、用户体验与引导设计
 
 ### 6.1 防迷路三大机制
 
 1. **永远有"下一步"按钮**：玩家不知往哪走时，系统可指引
 2. **永远有"回贝克街"按钮**：迷路了就回家，福尔摩斯会整理思路
 3. **进度指示器**：案件顶部显示线索收集与推理完成进度
 
 ### 6.2 首次使用引导（三步遮罩式）
 1. 告诉你"你在这"
 2. 告诉你"要去那"
 3. 告诉你"点击地点可以查看详情"
 
 只在第一次显示，之后不再打扰。

 ## 七、分阶段发布策略
 
 ### 7.1 第一阶段：网页游戏（优先）
 
 **核心目标：** 降低玩家进入门槛，打开即玩
 
 - 网页游戏为主，玩家无需下载安装，直接在浏览器中打开即可体验
 - 支持游客身份进入游戏体验，零注册门槛
 - 游客身份**不能保存进度**
 - 当游客玩家想保存游戏进度时，弹出注册窗口引导注册
 
 **注册方式：**
 
 | 注册方式 | 验证机制 | 说明 |
 |----------|----------|------|
 | 用户名注册 | 密码 | 最快捷，适合想快速注册的玩家 |
 | 手机号注册 | 短信验证码 | 国内主流方式，便于找回 |
 | 邮箱注册 | 邮箱验证码 | 国际通用，海外玩家友好 |
 
 ### 7.2 第二阶段：手机应用端
 
 **触发条件：** 网页端游戏玩家积累到一定数量后
 
 - 开发手机应用端，获得更好的游戏体验（性能、通知、离线等）
 - **手机应用与网页版共用同一后端**，保持数据一致性
 - 玩家在任何平台注册的账号，均可跨平台登录，进度无缝同步
 
 ### 7.3 后端架构原则
 
 - **单一后端**：Web端和移动端共用同一套后端API
 - **数据一致性**：无论从哪个平台访问，玩家数据完全一致
 - **渐进增强**：Web端保证核心功能可用，移动端提供增强体验

 ## 八、技术实现方案
 
 **技术选型：**
 - **游戏引擎：** Godot 4.7（锁定）| Compatibility/GLES3 | GDScript | TileMapLayer
 - **导出策略：** Web优先（M1）→ 原生手机（M2+）
 - **前端通信：** M1仅HTTPRequest（信号驱动）→ M2+WebSocket（线索推送/跨设备同步）→ M3+多人。禁用HTTPClient
 - **后端：** RESTful+JSON，Web/Mobile共用API
 
 **存储方案：**
 - 游客：本地临时缓存（退出即失，引导注册后持久化）
 - 注册用户：本地 + 服务器同步
 
 **存档同步策略：** M1服务端覆盖（最新时间戳为准）→ M2+案件级合并 → M3+双向+手动选择
 
 **瓦片加载：** 案件Rect2按需加载，单TileMapLayer+set_cell()动态管理，淡入0.3s过渡
 
 **项目目录（res://）：** scenes/(systems|cases|ui) · scripts/(systems|cases|ui) · cases/(.tres/.res) · assets/(tilesets|sprites|fonts) · data/(dialogue|config) · addons/
 
 **双版本号：** 客户端vX.Y.Z(功能) / 内容数据(案件/资源) / 存档格式(save_version)，三者独立

 ## 九、技术设计原则
 
 ### 🚨 两大核心原则
 
 **1. 稳定性**
 - 选用当前公认的稳定、可靠技术架构
 - 不能用着用着就崩溃，让人体验很不好
 - 技术选型：Godot 4.x + GDScript + TileMapLayer
 
 **2. 可扩展性**
 - 功能能够随着后续新的需求或认知进行扩展
 - 不要每一次扩展就要进行一次底层的重构
 - 案件库架构、模块化场景系统、数据驱动设计
 
 ### 关键设计准则
 
 | 错误做法 | 正确做法 |
 |---------|---------|
 | ❌ "你自己在地图上找劳瑞斯顿花园街在哪" | ✅ "点击这里直接前往劳瑞斯顿花园街" |
 | ❌ "这个地方有线索，你自己找吧" | ✅ "这个房间有3个可疑的地方，点击高亮的地方查看" |
 | ❌ "你自己想下一步该去哪" | ✅ "根据你的推论，下一步应该去见那个巡警" |
 
 > 玩家来玩这个游戏，是来体验当福尔摩斯破案的爽感，不是来研究1864年伦敦地理的。地图是收藏品，不是考题。

 ## 十二、测试策略
 
 > 推理游戏的测试比普通游戏更复杂——不仅要测功能是否正常，还要验证推理逻辑是否自洽、线索链路是否完整、难度曲线是否合理。测试不是开发结束后的事，而是贯穿整个开发过程。
 
 ### 12.1 测试原则
 
 - **内容正确性优先**：推理游戏的致命缺陷不是画面粗糙，而是推理逻辑不通、线索链断裂、时间锁死局
 - **玩家视角测试**：每个功能测试都要从"玩家体验"出发，而非"技术实现正确"
 - **难度模式全覆盖**：同一内容必须在简单/普通/困难三种模式下分别验证，确保各模式体验符合设计预期
 - **自动化与人工结合**：功能回归用自动化，内容体验和推理感受必须人工测试
 
 ### 12.2 测试维度
 
 | 测试维度 | 核心关注点 | 说明 |
 |----------|-----------|------|
 | **功能测试** | 各系统功能是否正确运行 | 推理墙操作、侦破工具使用、线索发现与计分、时间线推进、竞技场景交互等 |
 | **内容测试** | 推理链是否完整自洽 | 每条推理链的三步结构（发现→推断→验证）是否可达、线索是否能在预期场景被发现、推理墙关联是否正确 |
 | **时间锁测试** | 软锁机制是否有效 | 每个时间锁的 bypass_methods 和 remedy_methods 是否真实可用，是否存在死局 |
 | **难度测试** | 三模式体验差异化 | 简单模式引导是否充分、困难模式无提示是否可通关、普通模式概率触发是否自然 |
 | **兼容性测试** | 多平台多设备表现 | Web端：Chrome/Firefox/Safari/Edge；移动端：iOS/Android主流机型；不同分辨率适配 |
 | **性能测试** | 运行流畅度 | 瓦片加载速度、场景切换帧率、内存占用（尤其多场景叠加后）、Web端首次加载时间 |
 | **存档测试** | 数据安全与同步 | 游客→注册转化、本地存档保存/加载、跨平台数据同步、存档版本兼容性 |
 | **引导测试** | 新手体验 | 首次使用引导三步遮罩是否清晰、防迷路三机制是否生效、玩家能否在3分钟内理解核心玩法 |
 | **回归测试** | 版本更新稳定性 | 每次版本迭代后，核心功能（推理墙/线索/时间线/存档）是否正常，已有存档是否兼容 |
 
 ### 12.3 各阶段测试重点
 
 **MVP阶段：**
 - 推理墙5页签的拖拽/分组/删除功能正确性
 - 《血字的研究》14条推理链完整可达性（每条链的三步结构全部验证）
 - 三种难度模式在《血字的研究》中的差异化体验
 - 时间锁的 bypass/remedy 路径全部走通
 - 游客模式→注册→保存进度的完整流程
 - Web端主流浏览器兼容性
 - 首次引导流程可用性
 
 **案件扩展阶段：**
 - 新案件推理链与已有案件的逻辑自洽性
 - 多案件瓦片区域叠加后的地图性能
 - 连锁案件伏笔是否正确触发和衔接
 - 跨案件线索引用是否有效（双向锁校验）
 - 存档格式版本升级兼容性
 
 **发布前：**
 - 全平台全浏览器兼容性矩阵测试
 - 性能压测（极端场景：全部瓦片加载、推理墙满载）
 - 存档数据迁移测试（旧版本→新版本）
 - 安全测试（用户输入校验、接口鉴权、防作弊）
 - 玩家体验测试（邀请目标用户试测，收集反馈）
 
 ### 12.4 推理链专项验证
 
 > 推理游戏的核心资产是推理链，推理链的测试是整个测试策略中最重要的部分。
 
 每条推理链必须通过以下检查项：
 
 | 检查项 | 验证内容 | 通过标准 |
 |--------|---------|---------|
 | **可达性** | 推理链的每一步在对应难度模式下是否可达 | 简单/普通/困难均可走通 |
 | **线索可见性** | 线索是否在预期场景中被发现 | 不依赖bug或异常操作即可触发 |
 | **推断正确性** | 推理墙上的推断是否正确关联到线索 | 玩家按正确推理链操作后，推断结果与设计一致 |
 | **时间锁安全性** | 涉及时间锁的推理链是否有可用的 bypass/remedy | 错过时间窗口后至少1种方法可继续 |
 | **难度适配性** | 不同难度下的提示/引导是否合理 | 简单模式有明确引导、困难模式无提示但可通关、普通模式概率自然 |
 | **星级评价准确性** | 1-3星评价条件是否可正确触发 | 不同表现对应正确星级，边界条件无歧义 |
 
 ### 12.5 自动化测试策略
 
 | 层级 | 工具/方法 | 覆盖范围 |
 |------|----------|---------|
 | **GDScript 单元测试** | GUT（Godot Unit Test） | 计分算法、星级评价计算、时间锁逻辑、概率提示算法 |
 | **场景集成测试** | 自动化脚本模拟玩家操作 | 线索发现→推理墙操作→推断生成的完整流程 |
 | **存档回归测试** | 存档快照对比 | 每次版本更新后验证存档加载/保存一致性 |
 | **CI/CD 流水线** | GitHub Actions | 每次提交自动运行单元测试、Godot项目校验、导出验证 |
 
 ### 12.6 测试数据管理
 
 - 每个案件维护一份**推理链测试用例表**，记录每条推理链的预期输入和预期输出
 - 每次内容修改（新增/修改推理链、调整线索、修改时间锁）必须同步更新测试用例
 - 存档测试使用**固定种子存档**（记录特定游戏状态的存档快照），确保回归测试可重复
 - 性能测试使用**基准场景**（如：劳瑞斯顿花园街全瓦片加载 + 推理墙满载），记录基准数据用于对比

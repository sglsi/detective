### 3.2 关键结构性缺口（已读源码确认）

> **🔴 缺口 A — 存档持久化是"空壳"**（已核实 `save_manager.gd` 第 236-271 行）
> - `_get_dialogue_progress / _get_clue_states / _get_unlocked_locations / _get_completed_milestones` 全部直接 `return {}` / `return []`。
> - `_restore_dialogue_progress / _restore_clue_states / ...` 全部为 `pass`（空操作）。
> - **后果**：云存档 API 测试全绿（上传/下载一个 blob 成功），但该 blob 不含任何真实进度/线索/对话/里程碑。玩家"存档→读档"后游戏状态不会恢复。这是**测试假绿**的典型——传输层通，业务层空。

> **🔴 缺口 B — 线索/案件数据解析未接入**（已核实 `clue_system.gd` / `case_manager.gd`）
> - `clue_system.gd` 的 `ClueData` 从 `.tres` 解析、总线索数从案件数据获取均为 TODO。
> - `case_manager.gd` 的案件列表扫描为 TODO。
> - **后果**：推理墙、关卡进度等依赖线索状态的功能，可能缺少可被消费的真实数据来源。

> **🟠 缺口 C — 场景热点占位绘制**（已核实 `scene_controller.gd` `_draw_placeholder`）
> - 8 幕热点均用 emoji + 文本（`🌳 命案现场外围…`）程序化绘制，未使用已导入的美术资源。
> - **后果**：P3 模块 10 完成了资产导入与 `TextureRect`/`Button`/`Label` 动态化能力，但场景渲染仍走占位逻辑，真实美术未真正上屏。

## 四、核心风险与缺口（按严重度）

| 严重度 | 缺口 | 影响 |

| 🟠 中 | 场景热点占位绘制（缺口 C） | 真实美术未上屏，视觉与"美术动态化"目标脱节 |

| 🟡 低 | 主菜单云存档检查/设置 UI/音频淡入未实现 | 体验收尾项 |

## 五、优化建议与方向（按优先级）

### P5-1 🔴 补全存档真实收集/恢复（最高优先级）
- 实现 `save_manager.gd` 的 `_get_*` / `_restore_*`：对接 `DialogueManager`（对话进度）、`ClueSystem`（线索状态）、`MapManager`（已解锁地点）、`GameManager`（里程碑）。
- 加一条 **GDScript 端到端存档测试**：保存→重启→断言状态恢复，杜绝"API 绿、业务空"的假绿。

### P5-2 🔴 接入线索/案件数据解析
- `clue_system.gd`：从 `.tres` 资源解析 `ClueData`、从案件数据取总线索数。
- `case_manager.gd`：扫描 `data/cases/` 构建可用案件列表，驱动关卡选择。
- 验证推理墙五态机/红线有真实数据驱动。

### P5-3 🟠 替换场景占位绘制为真实美术
- 将 `scene_controller.gd` 的 `_draw_placeholder` 改为 `TextureRect` + 已导入资源（美术目录已就绪），接入 P3 模块 10 的动态化能力。
- 建立"热点配置 → 美术资源"映射表，避免逐幕硬编码。

### P5-4 🟠 内容深度打磨（承接 Phase 4）
- 场景 3：补齐六步 `exploration_step` 与 `verify_filter` 验证分支（或确认轻引导设计的等价机制）。
- 场景 7：补充华生/其他角色对话，改善角色互动厚度。

### P5-5 🟠 工程治理
- **远程版本控制**：`gh auth login` → `git remote add` → `git push`（GitHub 私有库）；引入 `main`/`dev` 分支或 trunk-based。
- **CI 加固**：加入定时复跑（cron）+ 新增 GDScript 单元测试套件（覆盖率真实游戏逻辑，而非仅 smoke/集成）。
- **文档治理**：校正路线图中的旧目录名，保持设计与代码命名一致。

### P5-6 🟡 体验收尾
- `main_menu.gd`：实现云端存档存在性检查与"继续游戏"入口、设置界面。
- `audio_manager.gd`：实现 fade_in/fade_out 过渡。

## 六、总体结论

当前项目**框架稳健、测试全绿、8 幕内容工程可达**，是一个"能跑通、能加载、结构合规"的扎实底座。但静态扫描揭示：**测试绿灯掩盖了两条未接好的业务主线——游戏状态持久化与线索/案件数据解析**。在补齐 P5-1 / P5-2 之前，游戏的"可玩性"停留在演示层面（进度不保存、推理缺数据）。

**下一步不是再加功能，而是把已搭好的管道真正接通**：先填存档与数据解析两个🔴缺口，再上真实美术与内容打磨。完成这三项，项目才从"工程可达"真正走向"可玩成品"。

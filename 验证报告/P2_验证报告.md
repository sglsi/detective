## 三、P2 交付物清单

### 3.1 对话资源（5 文件）

由 `tools/convert_dialogue_to_tres.py` 的参数化骨架 `_scene_autonomous_6steps()` 生成：

| 文件 | 节点数 | 说明 |
|------|:------:|------|
| `resources/dialogues/scene_04_police.tres` | 43 | 巡警兰斯叙述案发经过 → 选择关键假设 → 验证 → 线索汇总 |
| `resources/dialogues/scene_05_parlor.tres` | 40 | 会客厅勘查 → 老妇人证词 → 广告/戒指盒/逃逸马车 → 假设选择 → 验证 |
| `resources/dialogues/scene_06_apartment.tres` | 40 | 霍普公寓勘查 → 礼帽/照片/爱丽丝/电报 → 推理 → 验证 |
| `resources/dialogues/scene_07_hotel.tres` | 40 | 旅馆勘查 → 门血/药盒/钱包/铁丝 → 证据链闭合 → 验证 |
| `resources/dialogues/scene_08_finale.tres` | 39 | 终章 → 马车夫追捕/手铐/马车/霍普 → 案件完结 |

每个 `.tres` 均包含：
- **六步闭环**（`exploration_step` 1-6，7 个 `is_step_entry` 入口）
- **三难度分支**（`difficulty_filter` 0/1/2，EASY 含详细提示，NORMAL 含简要引导，HARD 仅问题本身）
- **四级验证分支**（VERIFIED / SUPPORTED / INSUFFICIENT / CONTRADICTORY，`verify_filter` 路由）
- **4 选项推理选择题**（`hypo_choice` 节点，4 个 `next_nodes`）
- **里程碑提示**（`note_text` / `stage_direction` 阶段性总结）

### 3.2 场景文件（10 文件）

| 文件 | class_name | 说明 |
|------|------------|------|
| `scripts/scene/scene_04.gd` | `Scene04Game` | 警局勘查（guidance_level=2, multi_observe=true） |
| `scripts/scene/scene_05.gd` | `Scene05Game` | 会客厅勘查 |
| `scripts/scene/scene_06.gd` | `Scene06Game` | 霍普公寓勘查 |

每个 `Scene0NGame` 均继承 `GameScene`，实现：
- `_setup_ui()` / `_connect_signals()` / `_start_tutorial()`（加载 scene_id + `.tres`）
- `_on_hotspot_clicked()`（自主探索，无进度阻断）
- `_on_verification_complete()`（四级结果转发 `advance_to`）
- `_on_scene_node_entered()`（终点检测）

### 3.4 推理墙重写

`scripts/clue/reasoning_wall_ui.gd`（~590 行，从骨架完整重写）：

| 组件 | 描述 |
|------|------|
| **线索四态机** | `ClueState` enum：UNDISCOVERED → COLLECTED → VERIFIED → EXPIRED |
| **假设五态机** | `HypothesisState` enum：PENDING → ACTIVE → LEADING → CONFIRMED / EXCLUDED |
| **红线推理链** | `RelationType` enum（SUPPORT/CONTRADICT/WEAK），`_draw()` 渲染绿实线/红虚线/灰虚线 |

| **四级验证** | `_verify_all()` → support≥3 且无矛盾 → VERIFIED → 主假设 CONFIRMED |
| **向后兼容** | `VerifyResult` enum 与 `ClueEventBus.verification_complete` 信号签名不变 |

## 四、校验中暴露并修复的真实缺陷

### 4.1 P2 新增缺陷

| # | 缺陷 | 位置 | 根因 | 修复 |
|---|------|------|------|------|
| 1 | `debug_exercise()` 中 `ring` 线索 `relation_tags` 包含 `h_cabman` → `_decide_relation` 返回 SUPPORT 而非 CONTRADICT，红线永远不触发 | `reasoning_wall_ui.gd:548-565` | `ring` 在 `_CASE_CLUES` 中 `rel` 设为 `["h_cabman"]`，优先级高于「干扰」判定 | 改用 `distractor_ring`（`rel: [], attr: ["干扰"]`），并改为两阶段演练：先 5 条支持验证 → VERIFIED+CONFIRMED，再加矛盾线索生成红线 |
| 2 | `_verify_all()` 返回的 `overall` 与 `_hypotheses[h_cabman].state` 在二次验证后不同步 — Phase 2 矛盾关联导致 `h.state` 被重置为 ACTIVE | `debug_exercise()` | 状态在 Phase 2 后被覆盖 | 在 Phase 1 后立即保存 `result1` 和 `state1`，Phase 2 仅用于生成红线 snapshot |

### 4.2 跨阶段历史遗留缺陷（在 P2 中暴露）

| # | 缺陷 | 位置 | 根因 | 修复 |
|---|------|------|------|------|
| 3 | `clue_event_bus.gd` 和 `dialogue_event_bus.gd` 未声明任何 `signal`，但多处 `emit_signal` + `connect` 使用未声明信号 | `autoload/clue_event_bus.gd`、`autoload/dialogue_event_bus.gd` | 原始代码仅 `extends Node` 无信号声明；Godot 允许 `emit_signal` 不声明但 `connect` 必须声明 | 为两个 EventBus 补充完整信号声明：`clue_discovered`、`clue_recorded`、`clues_linked`、`verification_complete`、`dialogue_trigger`、`dialogue_finished`、`node_entered`、`step_entered` |
| 4 | `ReasoningWallUI` 中 `to_local()` 在 `--script` 模式下编译失败 | `reasoning_wall_ui.gd:506-515` | `to_local()` 是 `CanvasItem` 方法，在 `SceneTree` 脚本上下文中 `load()` 编译阶段无法解析 | 替换为自定义 `_to_local(pos) -> pos - global_position`，并改为 `get_node_or_null("/root/ClueEventBus")` / `get_node_or_null("/root/UIManager")` 替代 autoload 直接引用，确保 `--script` 模式可编译 |

### 4.3 P2 校验工具自身缺陷

| # | 缺陷 | 位置 | 根因 | 修复 |
|---|------|------|------|------|
| 5 | `p2_wall_script_check.gd` 中 enum 数值预期错误（`VERIFIED=3` vs 实际 `VERIFIED=0`） | `p2_wall_script_check.gd` | GDScript enum 从 0 开始，校验脚本误以为从 1 开始 | 修正所有 enum 数值预期：`VerifyResult.VERIFIED=0`、`ClueState.VERIFIED=2`、`HypothesisState.CONFIRMED=3` |
| 6 | `_init()` 中 `await RenderingServer.frame_post_draw` 在 SceneTree 脚本中不执行 | `p2_wall_script_check.gd` | `SceneTree._init()` 不支持 `await` | 改为 `_process(_delta)` 驱动，frame 2 时执行校验 |
